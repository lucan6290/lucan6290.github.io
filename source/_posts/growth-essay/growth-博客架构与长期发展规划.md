## 前言

本文档为「箓川码笺」博客的长期发展宪章。

> **核心理念：稳定优于创新，规范优于灵活，可迁移优于定制化。**

---

## 一、博客定位与发展规划

### 1.1 博客定位

| 维度 | 定位 | 说明 |
|-----|-----|-----|
| **内容类型** | 技术为主，个人成长为辅 | 技术占 80%，个人成长占 20% |
| **目标读者** | 未来的自己，同行工程师 | 以个人沉淀为主，分享为辅 |
| **内容风格** | 严谨、详细、可复现 | 记录完整思考过程和踩坑记录 |
| **更新频率** | 无固定周期，有沉淀即发布 | 不追求热点，不追求日更 |

### 1.2 内容增长预估

| 时间 | 预计文章数 | 规模说明 |
|-----|---------|---------|
| 第1年 | 50-100篇 | 建立分类体系和内容框架 |
| 第5年 | 300-500篇 | 需要搜索、标签分类、归档 |
| 第10年 | 800-1500篇 | 可能需要内容重组、专题整理 |

## 二、分类体系设计

### 2.1 一级分类（5个，保持稳定）

| 一级分类 | 用途 | 创建目录 |
|---------|------|---------|
| **技术研习** | 系统的技术学习 | `source/_posts/tech-study/` |
| **踩坑复盘** | 问题解决记录 | `source/_posts/pitfall-review/` |
| **项目实战** | 完整项目相关 | `source/_posts/project-practice/` |
| **成长随笔** | 个人成长、感悟 | `source/_posts/growth-essay/` |
| **资源分享** | 工具、资源整理 | `source/_posts/resource-sharing/` |

### 2.3 分类选择决策树

```
开始写文章
    ↓
是关于「解决问题」吗？
    ├─ 是 → 踩坑复盘
    └─ 否
        ↓
是关于「完整项目」吗？
    ├─ 是 → 项目实战
    └─ 否
        ↓
是「分享资源」吗？
    ├─ 是 → 资源分享
    └─ 否
        ↓
是「个人成长/感悟」吗？
    ├─ 是 → 成长随笔
    └─ 否
        ↓
→ 技术研习
```

### 2.4 为什么不用三级分类？

| 问题 | 说明 |
|-----|------|
| 选择困难 | 写文章时要决定 3 个层级，决策成本高 |
| 层级僵化 | 未来想调整层级很麻烦（链接会变） |
| 维护成本 | 层级越深，维护成本越高 |

**推荐方案：2 级分类 + 标签 + 系列专题**

| 层级 | 组织方式 | 稳定性 |
|-----|---------|--------|
| 一级分类 | 固定 | 极高（一年以上不变） |
| 二级分类 | 固定 | 高（半年以上不变） |
| 标签 | 灵活 | 随时增减 |
| 系列专题 | 临时 | 有 3 篇以上相关文章时使用 |

## 三、系列专题组织

### 3.1 何时使用系列专题？

当有 3 篇以上紧密相关的文章时，使用 `series` 字段组织。

```yaml
---
title: Claude Code 入门指南
categories:
  - [技术研习, AI 探索]
tags: [claude-code, ai-agent]
series: "Vibe Coding 使用专题"
series_order: 1
---
```

### 3.2 系列导航

在每篇文章开头或结尾添加：

```markdown
**📚 本系列文章：**
1. [Claude Code 入门指南]()
2. [AI 项目开发规范]() ⬅️ 当前位置
3. [Claude Code 使用技巧]()
```

## 四、内容分级策略

### 4.1 三级状态

| 状态 | 值 | 说明 | 部署行为 |
|------|------|------|---------|
| 草稿 | `draft` | 刚开始写，内容不完整 | 部署时自动排除 |
| 半成品 | `wip` | Work In Progress | 显示 WIP 标识 |
| 正式文章 | `published` | 完成度高（默认值） | 正常发布 |

### 4.2 使用方式

```yaml
---
title: 文章标题
status: wip  # draft / wip / published（默认）
---
```

## 六、链接稳定性策略

### 6.1 URL 设计

当前配置：
```yaml
permalink: :year/:month/:day/:title/
```

**优势：**
- 语义化
- 便于归档
- 日期作为永久ID的一部分
- 未来迁移容易处理

### 6.2 链接不变承诺

一旦文章发布，URL 原则上**永不更改**。

```yaml
# 文件名可以改，但添加 slug 保持 URL
slug: 旧的url路径
```

## 七、文章元数据规范

### 7.1 完整模板

```yaml
---
title: 文章完整标题
date: 2026-05-23 10:00:00
updated: 2026-05-24 15:30:00

categories:
  - [一级分类, 二级分类]

tags:
  - 技术标签1
  - 技术标签2

description: 100-200字的文章摘要，清晰描述文章内容。

cover: /img/covers/xxx.svg    # 自动根据一级分类分配
sticky: 100                   # 置顶优先级（可选）
slug: 自定义URL路径            # 保持链接稳定（可选）
status: published             # draft / wip / published
---
```

### 7.2 标签使用规范

| 原则 | 说明 |
|-----|-----|
| 适中数量 | 3-8 个标签最佳 |
| 粒度适中 | 既不要太泛也不要太细 |
| 避免重复 | 不要同时出现意思接近的标签 |

## 八、长期内容组织策略

### 8.1 内容生命周期

```
想法 → 草稿 → 撰写中 → 发布 → 维护 → 归档
```

### 8.2 文章质量分级

| 级别 | 说明 | 比例预期 |
|-----|------|---------|
| **永久典藏** | 高质量、经得起时间考验 | 10% |
| **长期有效** | 几年内有价值 | 30% |
| **时效性内容** | 技术选型、版本特定 | 40% |
| **草稿/备忘** | 个人临时记录 | 20% |

## 九、维护与运营指南

### 9.1 年度检查清单

每 12 个月运行一次：

- [ ] 检查所有外部链接是否还可用
- [ ] 清理过时的内容（添加 deprecate 标记）
- [ ] 回顾分类体系是否需要调整
- [ ] 更新「年度总结」文章
- [ ] 备份所有 Markdown 源文件
- [ ] 检查依赖是否有安全更新

### 9.2 内容更新策略

- ✅ 小错误：直接编辑原文章
- ✅ 信息补充：添加"更新于 xxx"章节
- ❌ 重大变更：写新文章，旧文章添加过时标记

## 总结

建立这个规范的核心目的：

- 解放你在每个细节上做决策的精力
- 让博客结构清晰，即使几年后回来还能快速上手
- 让技术和内容分离，方便未来技术栈迁移
- 降低维护成本，让博客能长期陪伴你

> **记住：内容才是最重要的。规范是为了保护内容，而不是让你成为规范的奴隶。**
