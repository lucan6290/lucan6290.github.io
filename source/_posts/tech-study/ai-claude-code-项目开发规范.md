## 前言

AI 辅助开发不是「让 AI 替你写代码」，而是建立一套高效的人机协作模式。良好的规范能让 AI 产出更符合预期的代码，同时保持项目的可维护性。

本文档是「箓川码笺」使用 Claude Code 进行项目开发的标准规范。

---

## 一、协作模式：人机分工原则

### 1.1 开发者职责

| 职责 | 说明 |
|------|------|
| 🎯 **需求定义** | 明确产品功能、技术选型、架构设计 |
| 🏗️ **架构决策** | 确定技术栈、模块划分、核心流程 |
| ✅ **质量把控** | Code Review、测试验收、代码合并 |
| 📚 **知识积累** | 沉淀项目规范、业务逻辑、踩坑记录 |

### 1.2 AI 助手职责

| 职责 | 说明 |
|------|------|
| ⚡ **代码实现** | 根据需求编写具体代码、组件、函数 |
| 🔧 **工具使用** | 执行命令、运行测试、构建部署 |
| 📝 **文档编写** | 生成注释、README、API 文档 |
| 🐛 **辅助排查** | 分析错误日志、提供调试思路 |

### 1.3 协作流程

```
1. 开发者：提出需求 + 技术方案
2. AI：实现代码 + 询问确认
3. 开发者：Review 代码 + 提出修改
4. AI：迭代优化
5. 开发者：测试 + 合并
```

## 二、项目规范传达

### 2.1 项目初始化时的规范传达

在开始开发前，先向 Claude 介绍项目的整体情况：

**标准开场白模板：**

> "我们来开发一个 [项目类型] 项目，技术栈是 [技术栈]。
> 
> 项目规范：
> 1. [规范1]
> 2. [规范2]
> 3. ...
> 
> 先帮我搭建项目骨架。"

**示例：**

> "我们来开发一个 Hexo 博客，使用 Butterfly 主题。
> 
> 项目规范：
> 1. 文章文件名用 kebab-case
> 2. 中文为主，技术术语可用英文
> 3. 先了解现有结构再修改
> 4. 重大变更先确认再执行
> 
> 先帮我看看现有项目结构。"

### 2.2 .claude-guidelines.md 文件（强烈推荐）

在项目根目录创建 `.claude-guidelines.md`，Claude 会自动读取这个文件。

**模板见本文最后一节「附录」。**

## 三、代码质量规范

### 3.1 代码风格

- ✅ **遵循现有风格**：先看项目现有代码，保持一致
- ✅ **清晰的命名**：变量名、函数名语义化，不随意缩写
- ✅ **适当的注释**：复杂逻辑加注释，公共 API 加 JSDoc/TSDoc
- ✅ **单一职责**：一个函数/组件只做一件事

### 3.2 错误处理

- ✅ **不要忽略错误**：try-catch 要处理异常，不要空 catch
- ✅ **友好的错误信息**：错误信息要说明「是什么、为什么、怎么解决」
- ✅ **边界条件**：考虑空值、undefined、异常输入

### 3.3 安全性

- ✅ **不要提交密钥**：API Key、密码放在环境变量
- ✅ **输入验证**：用户输入要验证、过滤、转义
- ✅ **权限最小化**：AI 执行命令时，只给必要权限

## 四、Git 与版本控制

### 4.1 Commit 规范

使用 Conventional Commits 格式：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type 类型：**

| type | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | `feat(auth): add login page` |
| fix | Bug 修复 | `fix(api): handle timeout error` |
| docs | 文档变更 | `docs: update README` |
| style | 代码格式 | `style: format with prettier` |
| refactor | 重构 | `refactor: extract utils function` |
| test | 测试相关 | `test: add unit tests` |
| chore | 构建/工具 | `chore: update dependencies` |

### 4.2 AI 生成代码的提交策略

**推荐流程：**

1. AI 生成代码 → 2. 开发者 Review → 3. 修改完善 → 4. 手动提交

**不要让 AI 自动提交**，保持提交信息的语义化和可追溯性。

## 五、测试与验证

### 5.1 测试驱动开发（TDD）AI 版

```
1. 开发者：定义需求 + 验收标准
2. AI：先写测试用例
3. 开发者：确认测试用例
4. AI：实现功能代码
5. AI：运行测试，验证通过
```

### 5.2 测试覆盖要求

- ✅ **核心逻辑**：必须有单元测试
- ✅ **API 接口**：必须有接口测试
- ✅ **关键路径**：必须有 E2E 测试（如登录、支付）

### 5.3 AI 辅助测试

可以让 AI 做：

- 生成测试用例
- 生成 Mock 数据
- 分析测试覆盖率
- 辅助定位测试失败原因

## 六、文档规范

### 6.1 代码注释

**JSDoc 示例（JavaScript/TypeScript）：**

```javascript
/**
 * 用户登录
 * @param {string} email - 用户邮箱
 * @param {string} password - 用户密码
 * @returns {Promise<User>} 用户信息
 * @throws {AuthError} 认证失败时抛出
 */
async function login(email, password) {
  // ...
}
```

### 6.2 README 与文档

- ✅ **项目 README**：项目介绍、快速开始、技术栈
- ✅ **API 文档**：接口地址、参数、返回值
- ✅ **部署文档**：环境变量、部署步骤
- ✅ **架构文档**：模块关系、数据流、设计决策

AI 可以帮你写初稿，但最终需要开发者审核。

## 七、常见问题处理

### 7.1 AI 「幻觉」（生成不存在的 API）

**现象：** AI 使用了某个库不存在的方法。

**应对：**
1. 让 AI 查官方文档确认
2. 自己验证 API 是否存在
3. 告诉 AI 正确的用法

### 7.2 AI 生成的代码不符合预期

**应对：**
1. 明确指出哪里不对
2. 给出示例代码或参考
3. 拆分成更小的步骤

### 7.3 执行命令失败

**应对：**
1. 让 AI 分析错误日志
2. 检查权限、路径、环境
3. 需要时手动执行确认

## 八、附录：.claude-guidelines.md 模板

在项目根目录创建此文件，Claude 会自动读取：

```markdown
# 项目开发规范

## 项目概览
- 项目名称：箓川码笺
- 技术栈：Hexo + Butterfly
- 主要功能：个人技术博客

## 代码风格
- 语言：中文为主，技术术语可用英文
- 文件名：kebab-case（小写加短横线）
- 提交信息：Conventional Commits 格式

## 协作规则
1. 先了解现有代码结构再修改
2. 重大变更（如重构、删除文件）先确认再执行
3. 不要提交敏感信息（密钥、环境变量等）
4. 保持代码简洁，添加必要注释
5. 修改后运行相关测试验证

## 目录结构
source/
├── _posts/      # 博客文章
├── _drafts/     # 草稿
└── about/       # 关于页面

themes/
└── butterfly/   # Butterfly 主题（不要直接修改）

## 常用命令
- npm run server    # 本地预览
- npm run build     # 构建
- npm run clean     # 清理缓存

## 联系与反馈
有问题请提 Issue 或联系 lucan。
```
