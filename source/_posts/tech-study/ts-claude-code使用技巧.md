## 一、提示词工程：让 AI 更懂你

### 1.1 提示词的核心要素

一个好的提示词包含：

| 要素 | 说明 | 示例 |
|------|------|------|
| 📋 **任务** | 要做什么 | "创建一个组件" |
| 📍 **上下文** | 在哪里做 | "在 src/components/ 下" |
| 🎯 **要求** | 具体规格 | "用 Vue 3 + TypeScript" |
| 🎨 **风格** | 代码风格 | "参考现有代码风格" |
| ✅ **验收** | 怎么算完成 | "包含头像、姓名、邮箱" |

### 1.2 提示词模板

**基础模板：**

```
[任务]：在[位置]做[什么]
[技术栈]：使用[技术1]、[技术2]
[要求]：
1. [要求1]
2. [要求2]
[参考]：参考[某个文件/示例]
```

**示例：**

```
任务：在 src/components/ 创建 UserCard 组件
技术栈：Vue 3 + TypeScript + Composition API
要求：
1. 显示用户头像、姓名、邮箱
2. 添加编辑和删除按钮
3. 样式使用 Tailwind CSS
参考：参考 src/components/PostCard.vue 的风格
```

### 1.3 常用提示词技巧

#### 技巧 1：给 AI 一个角色

```
你是一位资深前端架构师，精通 Vue 3 最佳实践。
请帮我设计一个可复用的表单组件...
```

#### 技巧 2：分步执行

复杂任务拆成小步骤：

```
第一步：先分析项目结构，了解技术栈
第二步：设计组件架构
第三步：实现核心功能
```

#### 技巧 3：提供参考代码

```
参考这个组件的风格：
[粘贴一段代码]
```

#### 技巧 4：要求 AI 先思考再执行

```
先思考一下实现方案，确认后再写代码。
```

## 二、高效协作模式

### 2.1 任务拆解策略

**不要：**

```
帮我做一个完整的博客系统。
```

**要：**

```
1. 先搭建项目骨架
2. 再创建首页组件
3. 然后实现文章列表
4. 最后添加详情页
```

### 2.2 反馈循环

```
开发者：提出需求
    ↓
AI：实现 + 询问确认
    ↓
开发者：Review + 指出问题
    ↓
AI：迭代优化
    ↓（循环直到满意）
```

**Review 时的反馈技巧：**

- ❌ 不要只说「不对」
- ✅ 要说「这里应该是 xxx，因为 xxx，参考 xxx」

**示例：**

```
这里的错误处理不太对。
应该：
1. 先检查 response.ok
2. 再解析 JSON
3. 抛出具体的错误信息
参考 src/utils/request.js 的写法。
```

### 2.3 上下文管理

随着对话变长，AI 可能会「遗忘」之前的信息。

**应对：**

1. 定期总结当前状态
2. 把重要规范放在对话开头
3. 使用 `.claude-guidelines.md`
4. 必要时开新对话

## 三、项目开发进阶技巧

### 3.1 新项目初始化

**标准流程：**

```
1. 描述项目需求和技术栈
2. 让 AI 生成项目结构建议
3. 确认后开始搭建
4. 配置开发规范（ESLint、Prettier 等）
5. 创建第一个可运行版本
```

**提示词示例：**

```
我们要开发一个任务管理应用。
技术栈：React + TypeScript + Vite + Tailwind CSS + React Query

请先：
1. 建议一个合理的目录结构
2. 初始化项目
3. 配置 ESLint 和 Prettier
4. 创建一个简单的 Todo 示例验证环境
```

### 3.2 代码重构

**重构时的提示词：**

```
请帮我重构这个文件：
[粘贴代码]

目标：
1. 提高可读性
2. 提取可复用逻辑
3. 保持功能不变
4. 添加必要注释

重构前先告诉我你的方案，确认后再执行。
```

### 3.3 Bug 排查

**让 AI 帮你 Debug：**

```
我遇到了一个问题：[描述问题]

错误信息：
[粘贴错误日志]

相关代码：
[粘贴相关代码]

请帮我：
1. 分析可能的原因
2. 提供排查步骤
3. 给出修复建议
```

## 四、不同场景下的最佳实践

### 4.1 学习新技术时

```
我想学习 Vue 3 的组合式 API。
请帮我：
1. 创建一个简单的示例项目
2. 实现核心特性（ref、reactive、computed 等）
3. 每个特性加详细注释
4. 最后写一个总结文档
```

### 4.2 阅读开源项目时

```
这是一个开源项目的源码，帮我：
1. 分析项目结构和技术栈
2. 理解核心功能的实现
3. 画一个架构图（用 Mermaid）
4. 总结设计亮点
```

### 4.3 写算法题时

```
题目：[题目描述]

请帮我：
1. 先分析题目，讲清楚思路
2. 写出题解，加详细注释
3. 分析时间、空间复杂度
4. 如果有多种解法，都讲一下
```

## 五、避坑指南

### 5.1 常见陷阱

| 陷阱 | 表现 | 应对 |
|------|------|------|
| 🤖 **幻觉** | AI 编造不存在的 API | 让 AI 查文档，自己验证 |
| 🚀 **过度设计** | 代码太复杂，过度工程化 | 提醒 AI 「保持简单」 |
| 📝 **代码风格不一致** | 每个文件风格不同 | 提供参考，强调一致 |
| 🔒 **安全问题** | 提交密钥、硬编码密码 | 创建 .claudeignore |

### 5.2 什么时候不要让 AI 做

- ❌ 核心业务逻辑（自己要完全理解）
- ❌ 复杂的算法设计（AI 可能选不对方案）
- ❌ 安全相关的代码（自己要 Review 每一行）
- ❌ 数据库迁移（破坏性操作要慎之又慎）

## 六、提升效率的快捷键与命令

### 6.1 Claude Code 桌面端快捷键

| 快捷键 | 功能 |
|--------|------|
| `Ctrl/Cmd + N` | 新对话 |
| `Ctrl/Cmd + O` | 打开项目 |
| `Ctrl/Cmd + Shift + P` | 命令面板 |
| `Ctrl/Cmd + Enter` | 发送消息 |

### 6.2 常用命令

让 AI 帮你执行：

```
# 项目相关
- "运行 npm install"
- "启动开发服务器"
- "运行测试"
- "构建生产版本"

# Git 相关
- "查看 git status"
- "查看 git diff"
- "提交这次修改"

# 其他
- "格式化代码"
- "检查 lint 错误"
```
