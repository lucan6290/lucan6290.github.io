# 管理员后端接口文档

## 1. 文档说明

本文档描述 `admin/backend` 的 HTTP API。接口统一使用版本前缀：

```text
/api/v1
```

当前后端采用本地优先模式，主要读写：

- `site/docs`
- `site/blog`
- `site/sidebars.ts`
- `site/blog/authors.yml`
- `admin/backend/data/content-schema`
- `site/static`

状态说明：

- 已实现：当前代码已具备该接口，并且应完全符合本文档契约。
- 部分实现：当前代码已有基础接口，但字段、参数、响应或错误码仍需继续对齐本文档。
- 规划中：接口设计已确定，后续实现。

契约优先原则：

- 本文档是管理员后端 API 的主契约。
- 后端代码、OpenAPI、前端调用、测试用例都应向本文档对齐。
- 如果当前代码或 OpenAPI 与本文档不一致，视为后端待修复问题，而不是前端兼容依据。

## 2. 通用约定

### 2.1 Base URL

本地开发默认：

```text
http://127.0.0.1:18000
```

接口完整示例：

```text
GET http://127.0.0.1:18000/api/v1/articles
```

### 2.2 Content-Type

除文件上传接口外，请求与响应均使用：

```http
Content-Type: application/json
```

文件上传接口使用：

```http
Content-Type: multipart/form-data
```

### 2.3 文章类型枚举

```text
docs
blog
```

### 2.4 文章 ID

`article_id` 由后端生成，不直接暴露 Windows 文件路径。

编码来源示例：

```text
docs:intro.md
docs:project-practice/site-build.md
blog:2026-06-29-site-rebuild.md
```

编码方式：URL-safe Base64，去除末尾填充符 `=`。当前实现的解码逻辑会自动补齐填充符。

前端不需要自行解析 `article_id`，只需要在详情、编辑、删除、图片等接口中原样传回。

### 2.5 成功响应策略

成功响应直接返回接口声明的 DTO 或数组，不额外包装 `{success, data}` 信封。

默认成功状态码：

| 方法 / 场景 | 默认状态码 |
| --- | --- |
| `GET` 查询 | `200 OK` |
| `PUT` 保存 | `200 OK` |
| `DELETE` 删除 | `200 OK` |
| `POST` 创建资源 | `201 Created` |
| `POST` 动作型接口 | `200 OK` |
| 启动异步任务 | `202 Accepted` 或 `200 OK`，以具体接口为准 |

说明：

- 手写接口文档以业务契约为准。
- FastAPI 自动生成的 OpenAPI 是当前代码快照，可通过 `/docs` 和 `/openapi.json` 查看。
- 如果 OpenAPI 与本文档不一致，后端实现必须调整到本文档契约。

### 2.6 认证与鉴权

当前 Admin 后端采用本地优先模式，默认仅在本机或受信任局域网运行，现阶段不强制认证。

后续生产化预留以下方案：

- 本地开发：允许关闭认证。
- 受控访问：支持 Bearer Token 或 Session Cookie。
- 危险操作：即使已认证，仍必须遵守 `dry_run` / `confirm` 约定。

### 2.7 统一错误响应

所有业务错误和请求参数错误统一返回：

```json
{
  "code": "article_not_found",
  "message": "文章不存在。",
  "request_id": "fbd64a1a3d704e77b1e6d6f36ccda88b",
  "details": {}
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | string | 机器可读错误码 |
| `message` | string | 面向用户或前端的错误描述 |
| `request_id` | string/null | 请求追踪 ID，同时写入响应头 `X-Request-ID` |
| `details` | object | 附加错误详情 |

常见 HTTP 状态码：

| 状态码 | 含义 |
| --- | --- |
| `200` | 请求成功 |
| `201` | 创建成功 |
| `400` | 请求业务参数不合法 |
| `404` | 资源不存在 |
| `409` | 资源冲突，例如目标文章已存在 |
| `422` | 请求体或查询参数类型校验失败 |
| `500` | 未预期服务端错误 |

通用错误码：

| HTTP | code | 场景 |
| --- | --- | --- |
| `422` | `request_validation_error` | FastAPI / Pydantic 请求参数校验失败 |
| `500` | `internal_error` | 未预期服务端错误 |

各接口错误码表只列该接口特有或业务上需要前端单独处理的错误码；通用错误码默认所有接口都可能返回。

### 2.8 分页约定

列表接口默认支持分页。当前已实现接口可先返回全量数组，后续实现分页时必须遵循本节契约，避免前端二次迁移。

分页查询参数：

| 参数 | 类型 | 必填 | 默认值 | 限制 | 说明 |
| --- | --- | --- | --- | --- | --- |
| `page` | integer | 否 | `1` | `>= 1` | 当前页码 |
| `page_size` | integer | 否 | `20` | `1 - 100` | 每页数量 |

分页响应模型：

```json
{
  "items": [],
  "page": 1,
  "page_size": 20,
  "total": 0,
  "has_next": false
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `items` | array | 是 | 当前页数据 |
| `page` | integer | 是 | 当前页码 |
| `page_size` | integer | 是 | 每页数量 |
| `total` | integer | 是 | 总数量 |
| `has_next` | boolean | 是 | 是否还有下一页 |

全文检索类接口允许使用 `limit` / `offset`，但响应必须返回 `total`，便于前端展示结果数量和继续加载状态。

### 2.9 排序约定

列表接口统一使用 `sort` 参数表达排序。

```text
sort=-updated_at
sort=title
```

规则：

- 字段名前加 `-` 表示降序。
- 不加 `-` 表示升序。
- 多字段排序使用英文逗号分隔，例如 `sort=-date,title`。
- 接口只允许使用各自文档中列出的排序字段。

通用错误：

| HTTP | code | 场景 |
| --- | --- | --- |
| `400` | `invalid_sort_field` | 使用了不被接口支持的排序字段 |

### 2.10 危险操作约定

删除、移动、批量同步、部署等有副作用的接口统一支持“预演优先”。

通用字段：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `dry_run` | boolean | 否 | `true` | 是否只返回影响计划，不实际执行 |
| `confirm` | boolean | 否 | `false` | 是否确认执行实际变更 |

规则：

- `dry_run=true` 时，不允许实际修改文件。
- `dry_run=false` 且 `confirm=false` 时，后端应返回 `400 confirmation_required`。
- 前端执行危险操作时，应先调用 `dry_run=true` 展示计划，再由用户确认后传 `dry_run=false, confirm=true`。

通用变更计划模型：

```json
{
  "dry_run": true,
  "requires_confirmation": true,
  "changes": [
    {
      "action": "update",
      "target": "site/sidebars.ts",
      "description": "追加 docs 文章 ID"
    }
  ],
  "warnings": []
}
```

响应规则：

- `dry_run=true` 时，返回 `MutationPlanDTO`，只描述将要发生的文件变更。
- `dry_run=false, confirm=true` 时，如果接口文档没有声明专用执行结果，也返回 `MutationPlanDTO`，其中 `dry_run=false`、`requires_confirmation=false`。
- 如果执行后需要返回任务或资源详情，具体接口会单独声明 `TaskDTO`、`ArticleDetailDTO` 或专用结果 DTO。

### 2.11 时间格式约定

时间统一使用 ISO 8601 字符串，并显式包含时区。

推荐：

```text
2026-06-29T18:00:00+08:00
```

规则：

- blog `date` 至少必须以 `YYYY-MM-DD` 开头。
- 后端默认时间使用 `Asia/Shanghai`。
- 响应中的任务时间字段使用 `started_at`、`finished_at`。

### 2.12 文件名、slug 与路径规则

文章创建与移动接口中的 `slug` 字段在本系统中表示文章文件名片段。该字段允许中文文件名，不再强制要求英文 slug。

`slug`、`target_slug`、`category_path[]`、`target_category_path[]` 均按单个路径片段校验。

说明：

- 允许中文、英文、数字、连字符、下划线等普通文本字符。
- 不能为空，不能使用 `.` 或 `..`。
- 不能包含路径分隔符 `/`、`\`。
- 不能包含 Windows 非法文件名字符：`<`、`>`、`:`、`"`、`|`、`?`、`*`。
- 不能包含控制字符，不能以空格或点号结尾。
- `category_path` 是字符串数组，每个数组项都必须满足上述路径片段规则。
- 图片上传的 `slug` 使用安全文件名片段规则，可保留中文文章名，见 2.13。

### 2.13 文件上传约定

图片上传接口统一限制：

| 项 | 规则 |
| --- | --- |
| 支持扩展名 | `.png`、`.jpg`、`.jpeg`、`.webp`、`.gif`、`.svg` |
| 单文件大小 | 默认不超过 `10MB` |
| 文件名规则 | 使用安全文件名片段；可包含中文、英文、数字、连字符和点号，不能包含路径分隔符、控制字符或 Windows 非法文件名字符 |
| 重名处理 | 自动追加 `-1`、`-2` |
| 存储位置 | 文章同目录 `{文章文件名}-imgs` 文件夹 |
| 内容校验 | 必须校验扩展名、MIME 类型和文件 magic number；SVG 需要额外做脚本与外链清理 |

通用上传错误：

| HTTP | code | 场景 |
| --- | --- | --- |
| `400` | `unsupported_file_type` | 文件类型不允许 |
| `400` | `file_too_large` | 文件超过大小限制 |
| `400` | `invalid_file_name` | 文件名无法生成安全 slug |

### 2.14 限流约定

当前本地 Admin 不强制限流。后续接入 AI 写作、网页抓取、部署等高成本能力时，应按接口类型启用限流。

建议响应头：

| 响应头 | 说明 |
| --- | --- |
| `X-RateLimit-Limit` | 当前窗口请求上限 |
| `X-RateLimit-Remaining` | 当前窗口剩余次数 |
| `X-RateLimit-Reset` | 限流窗口重置时间 |

通用限流错误：

| HTTP | code | 场景 |
| --- | --- | --- |
| `429` | `rate_limit_exceeded` | 请求超过限流策略 |

### 2.15 并发控制约定

保存文章必须支持乐观锁，避免多个编辑端覆盖同一文件。移动文章、删除文章等危险操作当前使用 `dry_run` / `confirm` 机制降低误操作风险，文件版本前置校验后续可通过 `expected_version` 扩展。

规则：

- 读取文章摘要与详情时返回文件 `version` 和 `updated_at`。
- 保存文章时必须传入 `expected_version`。
- `expected_version` 与当前文件版本不一致时返回 `409 version_conflict`。
- 管理员强制覆盖能力后续可通过单独的 `force` 字段设计，不能隐式覆盖。

### 2.16 任务状态约定

构建、预览、部署、索引等异步任务统一使用以下状态：

```text
pending
running
success
failed
cancelled
```

任务响应通用字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `task_id` | string | 是 | 任务 ID |
| `type` | string | 是 | 任务类型，例如 `build`、`deploy`、`index` |
| `status` | string | 是 | 任务状态 |
| `started_at` | string/null | 否 | 开始时间 |
| `finished_at` | string/null | 否 | 结束时间 |
| `exit_code` | integer/null | 否 | 子进程退出码 |
| `logs` | string | 否 | 任务日志 |
| `error` | object/null | 否 | 失败错误 |

## 3. 通用响应模型

### 3.1 ValidationIssueDTO

用于描述文章校验问题。

```json
{
  "code": "blog_date_missing",
  "message": "blog 文章建议填写 date。",
  "severity": "warning"
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `code` | string | 是 | 问题代码 |
| `message` | string | 是 | 问题说明 |
| `severity` | string | 是 | `error` 或 `warning` |

### 3.2 ArticleSummaryDTO

文章摘要，用于列表接口。

```json
{
  "id": "ZG9jczppbnRyby5tZA",
  "type": "docs",
  "type_label": "知识库",
  "title": "知识库说明",
  "description": "箓川码笺的长期知识库入口。",
  "relative_path": "intro.md",
  "route": "/docs/intro",
  "slug": "intro",
  "tags": [],
  "authors": [],
  "category_path": [],
  "category_label": "知识库",
  "sidebar_registered": true,
  "version": "mtime:1782746400.123",
  "updated_at": "2026-06-29T18:00:00+08:00",
  "issues": []
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | string | 文章 ID |
| `type` | string | `docs` 或 `blog` |
| `type_label` | string | 内容类型显示名，如 `知识库` / `博客` |
| `title` | string/null | 文章标题 |
| `description` | string/null | 文章描述 |
| `relative_path` | string | 相对 `site/docs` 或 `site/blog` 的路径 |
| `route` | string | Docusaurus 前端路由 |
| `slug` | string | docs 为 doc id，blog 为 Front Matter slug |
| `tags` | string[] | 标签列表 |
| `authors` | string[] | 作者 ID 列表 |
| `category_path` | string[] | 分类路径；docs 来自目录，blog 来自注册表匹配 |
| `category_label` | string | 分类显示名，优先来自 `admin/backend/data/content-schema/categories.yml` |
| `sidebar_registered` | boolean/null | docs 是否登记到 `sidebars.ts`；blog 为 null |
| `version` | string | 文件版本，用于乐观锁；建议由文件 mtime 和 size 生成 |
| `updated_at` | string | 文件最后修改时间，ISO 8601 格式 |
| `issues` | ValidationIssueDTO[] | 校验问题 |

### 3.3 ArticleDetailDTO

文章详情，在 `ArticleSummaryDTO` 基础上增加：

```json
{
  "id": "ZG9jczppbnRyby5tZA",
  "type": "docs",
  "type_label": "知识库",
  "title": "知识库说明",
  "description": "箓川码笺的长期知识库入口。",
  "relative_path": "intro.md",
  "route": "/docs/intro",
  "slug": "intro",
  "tags": [],
  "authors": [],
  "category_path": [],
  "category_label": "知识库",
  "sidebar_registered": true,
  "version": "mtime:1782746400.123",
  "updated_at": "2026-06-29T18:00:00+08:00",
  "issues": [],
  "frontmatter": {
    "title": "知识库说明",
    "description": "箓川码笺的长期知识库入口。",
    "sidebar_position": 1
  },
  "body": "# 知识库说明\n\n这里用于沉淀长期有效的技术笔记。",
  "raw_content": "---\ntitle: 知识库说明\n---\n\n# 知识库说明\n",
  "image_dir": null
}
```

新增字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `frontmatter` | object | Front Matter 字典 |
| `body` | string | 去除 Front Matter 后的 Markdown 正文 |
| `raw_content` | string | 文件完整原文 |
| `image_dir` | string/null | 文章图片目录，目录不存在时为 null |

### 3.4 ArticleCreateDTO

创建文章请求体。

| 字段 | 类型 | 必填 | 默认值 | 限制 | 说明 |
| --- | --- | --- | --- | --- | --- |
| `type` | string | 是 | 无 | `docs` / `blog` | 文章类型 |
| `title` | string | 是 | 无 | `1 - 120` 字符 | 中文标题 |
| `slug` | string | 是 | 无 | `1 - 120` 字符且符合路径片段规则 | 文章文件名片段，允许中文 |
| `description` | string/null | 否 | `null` | 最多 300 字符 | 文章描述 |
| `body` | string | 否 | `""` | 无 | Markdown 正文 |
| `category_path` | string[] | docs 建议 | `[]` | 每项符合路径片段规则 | docs 所属目录层级，允许中文 |
| `sidebar_position` | integer/null | 否 | `null` | `>= 1` | docs Front Matter 排序 |
| `authors` | string[] | blog 必填 | `[]` | 至少 1 项 | blog 作者 ID |
| `tags` | string[] | blog 必填 | `[]` | 至少 1 项 | blog 标签 |
| `date` | string/null | 否 | 当前上海时间 | 以 `YYYY-MM-DD` 开头 | blog 发布时间 |

### 3.6 ArticleMoveDTO

| 字段 | 类型 | 必填 | 默认值 | 说明 |

```json

}
```

docs 创建行为：

- 创建前会检查并创建 `site/docs` 下缺失的一级、二级分类目录。
- 缺失的分类会同步写入 `admin/backend/data/content-schema/categories.yml`，保持目录、注册表和接口返回一致。
- 文件写入 `site/docs/项目实战/开发规范/FastAPI 后端接口设计.md`。
- 自动生成 docs Front Matter。
- 自动计算文章 ID：`项目实战/开发规范/FastAPI 后端接口设计`。
- 如果目标分类在 `site/sidebars.ts` 中存在同级文章锚点，则自动追加到同级文章锚点后。
- 如果目标分类在 `sidebars.ts` 中没有同级文章锚点，会自动创建缺失的一/二级分类结构并写入文章 ID。
- 如果目标文章文件已存在，返回 `409 article_already_exists`，前端应提示用户更改文件名。

- 校验作者必须存在于 `site/blog/authors.yml`。
