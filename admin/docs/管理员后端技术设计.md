# 管理员后端技术设计

## 1. 设计目标

`admin/backend` 使用 Python FastAPI 实现，作为本地优先的博客与知识库管理后端。第一阶段目标不是做完整 CMS，而是把当前 Docusaurus 内容维护规则转成稳定的后端能力：

- 扫描 `site/docs` 和 `site/blog`。
- 解析 Markdown / MDX Front Matter。
- 识别 docs 与 blog 的不同规则。
- 校验必填字段、作者、侧边栏登记状态。
- 为创建、编辑、图片上传、构建预览提供统一接口。

## 2. 技术选型

- Web 框架：FastAPI
- 数据模型：Pydantic v2
- Front Matter 解析：PyYAML + 本地 Markdown 拆分逻辑
- 文件上传解析：python-multipart
- 文件系统：Python `pathlib`
- 存储方式：第一阶段不引入数据库，以本地文件为事实来源
- 运行方式：`uvicorn scr.main:app --reload`

## 3. 路径配置

后端默认从自身位置推导项目根目录：

```text
admin/backend/scr/core/config.py

```

核心路径：

- 项目根目录：`E:\A-Code\xiaocancoding`
- 站点目录：`site`
- docs 内容目录：`site/docs`
- blog 内容目录：`site/blog`

- content-schema：`admin/backend/data/content-schema`

后续可通过环境变量 `LUCHUAN_PROJECT_ROOT` 覆盖项目根目录。

## 5. 核心模型

### ArticleSummaryDTO

用于文章列表：

- `id`：后端生成的 URL 安全 ID。
- `type`：`docs` 或 `blog`。
- `title`：Front Matter 标题。
- `description`：Front Matter 描述。
- `relative_path`：相对内容目录的文件路径。
- `route`：Docusaurus 访问路由。
- `slug`：blog slug 或 docs 文档 ID。
- `tags`：blog 标签。
- `authors`：blog 作者。
- `category_path`：docs 分类路径。
- `sidebar_registered`：docs 是否登记到 `sidebars.ts`。
- `issues`：校验问题。

### ArticleDetailDTO

在 `ArticleSummary` 基础上增加：

- `frontmatter`
- `body`
- `raw_content`
- `image_dir`

## 6. 文章 ID 设计

文章 ID 不直接暴露 Windows 路径，而是将以下字符串进行 URL 安全 Base64 编码：

```text
docs:intro.md
docs:tech-study/java-interview/java-basic/java-vs-cpp.md
blog:2026-06-29-site-rebuild.md
```

这样前端可以用 `GET /api/v1/articles/{article_id}` 读取详情，同时避免路径分隔符干扰路由。

## 7. Front Matter 解析策略

第一阶段只支持标准 Markdown Front Matter：

```md
---
title: 示例
---

正文
```

解析原则：

- 文件开头不是 `---` 时，视为没有 Front Matter。
- Front Matter 使用 PyYAML 解析为 dict。
- YAML 解析失败时返回校验问题，不中断文章列表扫描。
- 读接口如实返回原文和拆分后的正文。

## 8. 校验规则

docs 校验：

- `title` 必填。
- `description` 建议填写。
- docs 文章 ID 应登记在 `site/sidebars.ts`。

blog 校验：

- `title` 必填。
- `slug` 必填。
- `authors` 必填。
- `tags` 必填。
- `description` 建议填写。
- `date` 建议填写。
- `authors` 必须存在于 `site/blog/authors.yml`。

第一阶段校验只报告问题，不自动修改文件。

## 9. 侧边栏读取策略

第一阶段以“读取和判断”为主：

- 从 `site/sidebars.ts` 中提取单引号包裹的 docs 文章 ID。
- 将 docs 文件路径转为文章 ID，例如：

```text
site/docs/project-practice/site-build.md
=> project-practice/site-build
```

- 判断文章 ID 是否存在于 sidebars。

后续写入 `sidebars.ts` 时，需要使用更稳妥的 TypeScript AST 或受控模板策略，避免粗暴字符串替换。

## 10. API 草案

第一阶段已实现：

- `GET /api/v1/health`：健康检查。
- `GET /api/v1/articles`：文章列表。
- `GET /api/v1/articles/{article_id}`：文章详情。
- `POST /api/v1/articles`：创建 docs 或 blog 文章。

创建文章接口约定：

- docs 请求需要提供 `type=docs`、`title`、`slug`、`category_path`，可选 `description`、`sidebar_position`、`body`。
- blog 请求需要提供 `type=blog`、`title`、`slug`、`authors`、`tags`，可选 `description`、`date`、`body`。
- `slug` 在创建文章接口中表示文章文件名片段，允许中文；`category_path` 片段也允许中文，但必须通过路径安全校验。
- docs 创建前会确保分类目录和 `admin/backend/data/content-schema/categories.yml` 分类注册表存在；创建成功后会同步 `site/sidebars.ts`。
- blog 创建成功后生成 `YYYY-MM-DD-slug.md`，并校验作者必须存在于 `site/blog/authors.yml`。
- 如果目标文章已存在，接口返回 409。
- 如果 docs 目标分类在 `sidebars.ts` 中没有同级文章锚点，会自动创建缺失的一/二级分类结构并写入文章 ID。

后续接口：

- `PUT /api/v1/articles/{article_id}`：保存文章。
- `DELETE /api/v1/articles/{article_id}`：删除或生成删除计划。
- `POST /api/v1/articles/{article_id}/validate`：单篇校验。
- `POST /api/v1/build`：执行 Docusaurus 构建。
- `POST /api/v1/preview/start`：启动预览。

## 11. 安全边界

所有文件操作必须满足：

- 只允许访问项目根目录下的白名单目录。
- 文章读取只允许访问 `site/docs` 和 `site/blog`。
- 禁止 `..` 路径穿越。
- API 不接收任意绝对路径作为读写目标。
- 上传图片时限制扩展名、大小、目标目录和安全文件名片段。
- 编辑器拖拽或粘贴图片时，前端只生成期望图片名和插入位置；后端仍负责创建 `{文章文件名}-imgs` 目录、校验图片内容并避免覆盖已有文件。
