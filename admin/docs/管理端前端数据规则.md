# 管理端前端数据规则

## 分类来源

管理端前端不维护固定一级分类列表。分类显示名、排序、启用状态和封面信息统一来自后端分类接口，后端读取 `admin/backend/data/content-schema/categories.yml`。

- 文章列表和仪表盘优先使用文章接口返回的 `category_label`、`category_path`。
- 分类选择器优先使用 `/api/v1/categories` 转换后的分类注册表。
- 页面组件不得再写死 `tech-study`、`pitfall-review` 等具体分类 slug。
- 新增分类应维护 `admin/backend/data/content-schema/categories.yml`，再由后端接口同步给前端。

## 状态来源

当前内容规范不要求文章 Front Matter 写入 `status`、`draft` 或 `published`。

管理端列表中的“状态”是校验状态，不是发布生命周期：

- `正常`：文章没有 error/warning 级校验问题。
- `有提醒`：文章存在 warning 级校验问题。
- `需处理`：文章存在 error 级校验问题。

前端统一使用 `displayStatus` / `displayStatusLabel` 展示状态。旧字段 `status` 仅用于兼容历史缓存和少量旧类型，不作为列表与仪表盘展示依据。

## Front Matter 写入

编辑器和新建文章流程不主动写入 `status`、`draft` 或 `published`。如果旧文章已包含这些字段，解析器可以保留并序列化，但不应作为当前内容管理规则推广。

默认作者 `lucan` 是当前个人写作配置，允许保留。

## 内容注册表索引

管理端新增“内容注册表”页面，用于查看 SQLite 管理索引状态和索引实体。

- `admin/backend/data/content-schema/*.yml` 与 Markdown 仍是内容源文件。
- `admin/backend/data/registry_index.sqlite3` 是可删除、可重建的查询索引。
- 前端只通过 `/api/v1/registry-index/*` 查看、搜索和重建索引，不直接写 SQLite。
- “YAML 原文”允许查看和保存注册表原文；保存前由后端解析校验。
- “表单维护”保存的是 YAML 条目列表，保存后自动重建 SQLite 索引。
- “差异检查”比较 YAML 注册表键与 SQLite 索引键，用于发现未注册标签、目录推导分类等差异。
