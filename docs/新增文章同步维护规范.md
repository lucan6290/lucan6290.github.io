# 新增文章同步维护规范

本文基于当前 `site` 目录中的 Docusaurus 代码检查结果编写，用于判断新增 `docs` 或 `blog` 文章时，需要同步修改哪些文件。

## 当前站点结构结论

检查到的关键文件：

- `site/docusaurus.config.ts`：配置 `docs`、`blog`、顶部导航栏和页脚入口。
- `site/sidebars.ts`：手动维护 `site/docs` 的左侧侧边栏。
- `site/blog/authors.yml`：维护博客作者 ID。
- `site/src/pages/index.tsx`、`site/src/pages/about.tsx`、`site/src/pages/projects.tsx`：当前只放通用入口，没有维护具体文章清单。

当前规则：

- `site/docs` 的路由前缀是 `/docs`。
- `site/blog` 的路由前缀是 `/blog`。
- `site/docs` 文章必须在 `site/sidebars.ts` 中登记，才会出现在左侧侧边栏中。
- `site/blog` 文章不需要维护侧边栏，Docusaurus 会按日期自动生成博客列表、归档和标签页。
- `site/docusaurus.config.ts` 中的顶部导航是手写的，只有希望出现在顶部菜单中的文章或分类才需要同步修改。
- `site/build` 和 `site/.docusaurus` 是构建产物，不手动修改。
- 当前没有使用 `_category_.json`，分类名称主要在 `site/sidebars.ts` 里维护。

## 通用命名规则

路径和文件名使用英文 slug：

```text
single-full-stack-efficient-workflow.md
java-vs-cpp.md
site-rebuild.md
```

页面标题、侧边栏标题和正文内容使用中文：

```md
---
title: 单人全栈开发高效流程
description: 适合单人全栈项目的高效交付流程。
sidebar_position: 1
---
```

`docs` 文章推荐最小 Front Matter：

```md
---
title: 文章标题
description: 文章说明
sidebar_position: 1
---
```

`blog` 文章推荐最小 Front Matter：

```md
---
slug: article-slug
title: 文章标题
authors: lucan
tags: [标签一, 标签二]
description: 文章说明
date: 2026-06-29T10:00:00+08:00
---
```

## 新增 docs 文章

### 情况一：在已有一级分类和已有二级分类中新建文章

示例：

```text
site/docs/tech-study/java-interview/java-basic/java-vs-cpp.md
```

需要修改：

1. 新建文章文件。
2. 修改 `site/sidebars.ts`，把文章 ID 加到对应二级分类的 `items` 中。

文章 ID 写法：

```ts
'tech-study/java-interview/java-basic/java-vs-cpp'
```

不用修改：

- `site/docusaurus.config.ts`，除非这篇文章要出现在顶部导航栏。
- `site/src/pages/index.tsx`、`site/src/pages/about.tsx`、`site/src/pages/projects.tsx`，当前这些页面没有维护文章清单。
- `site/build`、`site/.docusaurus`。

### 情况二：在已有一级分类下新建二级分类，并在二级分类里新建文章

示例：

```text
site/docs/dev-standards/
└── workflow/
    └── single-full-stack-efficient-workflow.md
```

需要修改：

1. 新建二级目录。
2. 新建文章文件。
3. 修改 `site/sidebars.ts`，在已有一级分类的 `items` 中新增一个 `category`。

示例：

```ts
{
  type: 'category',
  label: '流程与协作',
  collapsed: false,
  items: [
    'dev-standards/workflow/single-full-stack-efficient-workflow',
  ],
}
```

如果这个一级分类还没有对应的侧边栏结构，也需要先在 `site/sidebars.ts` 中补出一级分类。

可选修改：

- 如果希望顶部“知识库”菜单能直接进入这个新分类，需要修改 `site/docusaurus.config.ts` 的 `navbar.items`。

### 情况三：新建一级分类，再新建二级分类和文章

示例：

```text
site/docs/dev-standards/
└── workflow/
    └── single-full-stack-efficient-workflow.md
```

需要修改：

1. 新建一级目录。
2. 新建二级目录。
3. 新建文章文件。
4. 修改 `site/sidebars.ts`，新增一个 sidebar 分组。
5. 修改 `site/docusaurus.config.ts`，在顶部“知识库”下拉菜单中加入新一级分类入口。

`site/sidebars.ts` 示例：

```ts
devStandardsSidebar: [
  {
    type: 'category',
    label: '开发规范',
    collapsed: false,
    items: [
      {
        type: 'category',
        label: '流程与协作',
        collapsed: false,
        items: [
          'dev-standards/workflow/single-full-stack-efficient-workflow',
        ],
      },
    ],
  },
],
```

`site/docusaurus.config.ts` 示例：

```ts
{
  label: '开发规范',
  to: '/docs/dev-standards/workflow/single-full-stack-efficient-workflow',
}
```

说明：

- 新建一级分类时，最好同步给顶部导航加入口，否则用户可能只能通过已知 URL 或站内搜索进入。
- 当前页脚只放了“知识库”“博客”“项目”等总入口，一般不需要为每个一级分类改页脚。

### 情况四：移动已有 docs 文章到新分类

示例：把旧文章从：

```text
site/docs/project-practice/development-standards/single-full-stack-efficient-workflow.md
```

移动到：

```text
site/docs/dev-standards/workflow/single-full-stack-efficient-workflow.md
```

需要修改：

1. 移动文章文件。
2. 修改 `site/sidebars.ts`，删除旧文章 ID，加入新文章 ID。
3. 全站检查旧链接，如果旧路径出现在 `site/docusaurus.config.ts`、其他文章或页面中，要同步替换。

旧文章 ID：

```ts
'project-practice/development-standards/single-full-stack-efficient-workflow'
```

新文章 ID：

```ts
'dev-standards/workflow/single-full-stack-efficient-workflow'
```

## 新增 blog 文章

### 情况一：新增普通博客文章

示例：

```text
site/blog/2026-06-29-site-rebuild.md
```

需要修改：

1. 新建博客文章文件。
2. 确认 Front Matter 中的 `slug`、`title`、`authors`、`tags`、`description`、`date` 清晰可用。

示例：

```md
---
slug: site-rebuild
title: 重建个人技术站
authors: lucan
tags: [Docusaurus, 技术站, 知识库]
description: 记录个人技术站重建过程。
date: 2026-06-29T10:00:00+08:00
---
```

不用修改：

- `site/sidebars.ts`，博客没有使用 docs 侧边栏。
- `site/docusaurus.config.ts`，除非这篇文章要放进顶部“博客”下拉菜单。
- `site/src/pages/index.tsx`、`site/src/pages/about.tsx`、`site/src/pages/projects.tsx`，当前没有维护博客文章清单。
- `site/build`、`site/.docusaurus`。

### 情况二：新增博客文章，并希望出现在顶部“博客”菜单

需要修改：

1. 新建博客文章文件。
2. 修改 `site/docusaurus.config.ts`，在顶部“博客”下拉菜单中增加入口。

示例：

```ts
{
  label: '重建个人技术站',
  to: '/blog/site-rebuild',
}
```

注意：

- `to` 使用 `/blog/` 加 Front Matter 中的 `slug`。
- 顶部菜单只放重点文章，不建议每篇博客都加进去。

### 情况三：新增博客作者

如果 `authors` 不是已有的 `lucan`，需要修改：

```text
site/blog/authors.yml
```

示例：

```yml
newAuthor:
  name: 作者名
  title: 作者说明
  url: https://example.com
  image_url: https://example.com/avatar.png
```

然后文章里使用：

```md
authors: newAuthor
```

## 检查清单

新增 `docs` 文章后：

- 已创建 `.md` 或 `.mdx` 文件。
- 文件路径是英文 slug。
- Front Matter 至少包含 `title`。
- 已在 `site/sidebars.ts` 登记文章 ID。
- 如果是新一级分类，已考虑是否更新 `site/docusaurus.config.ts` 顶部“知识库”菜单。
- 如果移动了旧文章，已检查并替换旧链接。

新增 `blog` 文章后：

- 文件名建议使用 `YYYY-MM-DD-slug.md`。
- Front Matter 包含 `slug`、`title`、`authors`、`tags`。
- `authors` 已存在于 `site/blog/authors.yml`。
- 如果是精选文章，已更新 `site/docusaurus.config.ts` 顶部“博客”菜单。
- 不修改 `site/sidebars.ts`。

最终验证：

```bash
cd site
npm run build
```

如果构建报 broken links，优先检查：

- `site/sidebars.ts` 中的 docs 文章 ID 是否和真实路径一致。
- `site/docusaurus.config.ts` 中的 `to` 是否和文章路由一致。
- 文章内部 Markdown 链接是否仍指向旧路径。
