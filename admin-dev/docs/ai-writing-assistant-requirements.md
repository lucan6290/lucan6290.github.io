# AI 辅助写作页面需求文档

## 1. 背景

当前管理端已经具备文章创建、Markdown 编辑、Front Matter 编辑、图片上传、发布部署和基础 AI 写作助手能力。现有 AI 助手主要作为编辑器侧边栏使用，适合对已有内容进行润色、续写和生成片段，但还缺少一个面向“从想法到草稿”的完整写作工作台。

本需求要求在管理端单独增加一个“AI 辅助写作”页面，并保留编辑器内可展开/收拢的 LLM 写作侧边栏。功能分为三种模式，由用户手动选择：

1. Agent 模式：用户输入想法、片段、杂乱笔记、目的说明或非标准 Markdown 内容后，系统自动完成用户目的识别、必要求证、分类判断、二级分类判断或建议、文章标题建议、正文整理与 Markdown 草稿生成，并根据用户选择的权限模式确认或自动创建草稿。
2. LLM 模式：用户在编辑器中基于当前文章与 AI 对话，要求 AI 对文章内容执行插入、替换、删除、润色、续写、改写等操作。
3. 知识库稳定模式：用户基于现有本地文章进行问答，例如查询文章清单、统计分类数量、查找某篇文章内容、询问某个主题是否写过，并要求系统基于本地文章数据给出可追溯回答。

## 2. 范围

### 2.1 支持范围

- 不区分本地模式和在线模式，前端入口、交互流程、AI 工作流和用户体验保持一致。
- 仅处理博客文章写作、文章内容修改和基于现有文章的本地知识库问答。
- 本系统不涉及数据库，主要基于本地文件、Front Matter、文章目录和可重建的本地索引完成操作。
- 文章创建必须复用统一文章创建接口，由接口适配具体存储目标。
- 文章保存必须复用统一文章更新接口，由接口适配具体存储目标。
- AI 工作流必须使用 LangChain 或 LangGraph 编排；Agent 模式优先使用 LangGraph 表达固定步骤和状态流转。
- 新文章默认保存为草稿。
- AI 可以自主生成建议；请求批准模式下，涉及分类、命名、正文写入和图片占位时必须经过用户人工审核。
- 默认权限为“请求批准”；在该权限下，Agent 模式创建草稿前必须经过用户预览确认，LLM 模式修改文章前必须经过用户采纳确认。
- 用户可以在发送对话前选择更高授权，但授权只对当前消息或当前任务生效，不应成为不可见的全局永久授权。

### 2.2 不支持范围

- 不自动发布文章。
- 不自动提交 Git。
- 不直接修改 `themes/butterfly/`。
- 不绕过现有文章命名、分类、Front Matter 和创建参数规范。
- 不允许 AI 自行发明未注册的一级分类创建规则。
- 不引入必须长期运行的数据库服务。
- 不要求为了知识库问答引入独立向量数据库；如后续需要语义召回，只允许使用可删除、可重建的本地索引或嵌入缓存文件。

## 3. 用户入口

### 3.1 独立页面入口

管理端侧边栏增加“AI 辅助写作”菜单项。

用户点击后进入独立写作页面。页面默认展示模式切换控件，用户可在以下模式间手动切换：

- Agent 模式
- LLM 模式
- 知识库稳定模式

### 3.2 编辑器入口

编辑器页面保留 AI 写作助手侧边栏。

侧边栏默认可收起。用户需要时可以展开，用自然语言要求 AI 修改当前文章内容。

### 3.3 发送前权限选择

Agent 模式输入框和 LLM 模式对话输入框在发送前都需要提供权限选择。默认选中“请求批准”。

权限选项：

1. 请求批准
2. 替我审批
3. 完全访问权限

权限只控制 AI 写作助手在当前对话或当前任务中的自动执行范围，不改变系统全局权限、用户登录权限或后端接口鉴权。

权限枚举建议：

```ts
type AIApprovalMode = 'request-approval' | 'delegate-approval' | 'full-access'
```

三种权限对应的放行范围：

| 权限 | 含义 | 默认放行 | 必须再次请求用户确认 |
| --- | --- | --- | --- |
| 请求批准 | AI 只能生成方案、预览、问题和建议，所有写入动作都等用户点确认 | 读取当前文章、读取分类注册表、生成草稿方案、生成编辑建议、展示预览、提出求证问题 | 创建文章、写入正文、替换内容、删除内容、全文重写、修改 Front Matter、改分类、改标题、保存草稿 |
| 替我审批 | 用户授权 AI 在本次任务内自动采纳低风险结果，相当于让 AI 代点“确认/采纳” | 在 schema 校验通过、无高风险提示、无未回答求证问题时，可以自动创建草稿、写入正文、插入内容、替换选区、更新标题或 description 等低风险 Front Matter 字段 | 删除内容、全文重写、改一级分类、改二级前缀、低置信度方案、存在 `分类不确定`/`事实待验证`/`信息不足`/`可能包含敏感信息` 风险、图片占位无法确认 |
| 完全访问权限 | 用户授权 AI 在 AI 写作助手边界内连续执行完整写作任务，不逐步打断 | 在当前文章或当前 Agent 任务内，可以自动完成创建草稿、写入正文、插入、替换、删除正文片段、全文重写、更新允许的 Front Matter 字段、根据校验结果重试一次生成或写入 | 发布文章、部署站点、提交或推送 Git、删除文章文件、删除资源文件、修改主题源码、创建未注册一级分类、修改系统配置、执行命令行、越过后端接口直接写文件 |

权限边界要求：

- 三种权限都不能绕过后端接口鉴权和业务校验。
- 三种权限都不能创建未注册的一级分类。
- 三种权限都不能自动发布文章、提交 Git、推送代码、部署站点、删除文件或修改 `themes/butterfly/`。
- “替我审批”和“完全访问权限”也必须保留操作日志、生成摘要和回滚提示。
- 存在高风险、低置信度、求证未完成、图片占位不明确或 schema 校验失败时，即使用户选择更高权限，也必须降级为“请求批准”。
- 用户在同一任务中切换权限后，只影响切换后的下一次发送或后续自动步骤，不追溯已经生成的结果。

## 4. Agent 模式需求

Agent 模式用于“从想法到草稿”，也用于平时随手记录、整理笔记、沉淀思考和把零散素材转成可继续编辑的博客内容。它不是一次性替用户完成所有写作，而是把用户的原始输入稳定地整理成“可保存、可继续改、符合博客规范”的草稿方案；分类、标题、正文和图片占位由 AI 先提出建议，再根据用户选择的权限模式人工审核或自动写入。

用户输入可以是：

- 一句话想法
- 一段杂乱笔记
- 非标准 Markdown 内容
- 从别处复制来的粗糙文章
- 主题关键词
- 文章大纲
- 临时灵感
- 今日学习记录
- 问题排查过程
- 项目开发日志
- 读书摘录和个人理解
- 面试复盘和求职记录
- 待整理的资料链接或要点
- 对某个技术、工具、现象的短思考

系统需要基于输入内容完成固定工作流。工作流中的分类判断、标题生成、正文优化由 AI 自主决策，但整体步骤固定。

Agent 模式必须优先保证稳定、可控和可追溯：

- 不追求一次生成最终成稿，优先生成可继续编辑的高质量草稿。
- 不因为输入很短就编造细节。
- 不因为输入很乱就丢弃原始信息。
- 不因为分类不确定就创建未注册分类。
- 不在权限允许和校验通过前创建或修改文章文件。
- 不把 AI 判断当成事实，必须保留判断理由和不确定项。

### 4.0 使用场景与处理目标

Agent 模式需要覆盖以下常见写作场景：

| 场景 | 用户输入特点 | Agent 处理目标 |
| --- | --- | --- |
| 灵感速记 | 一句话、几个关键词、临时想法 | 生成短草稿，保留核心想法，列出待补充问题 |
| 学习笔记 | 学习过程、概念记录、代码片段 | 整理为“背景、概念、实践、总结”结构 |
| 踩坑复盘 | 环境、报错、尝试过程、最终方案 | 整理为“环境、现象、原因、解决、注意事项”结构 |
| 项目日志 | 功能实现、设计取舍、开发记录 | 整理为“目标、方案、实现、问题、结果”结构 |
| 思考随笔 | 观点、感受、判断、反思 | 整理为“触发、观点、论据、反思、后续行动”结构 |
| 资料摘录 | 链接、摘抄、零散要点 | 整理为“资料来源、要点、理解、可延伸问题”结构 |
| 粗糙文章 | 已有段落但结构混乱 | 重排结构，补齐标题层级，保留原意 |
| 文章大纲 | 已有标题和小节 | 扩写为可继续编辑的正文草稿 |

Agent 应自动识别输入更接近哪种场景，并据此选择正文结构。页面不提供独立的写作目标下拉框；如果用户在输入中自然说明写作目标，则以用户表达为优先。

可选写作目标：

- 自动识别
- 博客草稿
- 学习笔记
- 踩坑复盘
- 项目日志
- 思考随笔
- 资料整理
- 大纲扩写

不同写作目标的生成要求：

- 博客草稿：结构完整，适合后续发布前继续润色。
- 学习笔记：强调概念解释、理解路径和示例。
- 踩坑复盘：强调环境、现象、排查过程和最终方案。
- 项目日志：强调目标、实现过程、取舍和结果。
- 思考随笔：强调真实观点、推理链路和后续问题。
- 资料整理：强调来源、摘要、个人理解和待验证点。
- 大纲扩写：保留用户原有大纲，不随意改变主题。

### 4.1 Step 1：用户输入内容

页面展示一个主要输入框。

输入框支持多行文本。占位提示示例：

```text
写下你的想法、笔记、素材或文章草稿，AI 会整理成博客草稿。
```

输入区应支持用户随时记录，不要求一次写完整。输入体验要求：

- 支持较短输入，例如一句话或几个关键词。
- 支持较长输入，例如完整笔记或粗糙文章。
- 支持粘贴 Markdown、普通文本、代码块、报错信息和链接。
- 支持用户用自然语言说明本次目的，例如“整理成规范笔记”“保留图片位置占位”“写成踩坑复盘”“先生成草稿不要扩写太多”。
- 支持通过加号上传 `.md`、`.markdown`、`.txt`、`.text`、`.log`、`.docx` 素材文件；文件内容作为本次写作上下文并入输入，不自动保存为文章文件。
- 支持用户在发送前选择权限，默认“请求批准”。
- 不提供写作目标、目标读者、期望篇幅、写作风格等独立参数表单；这些要求只从用户输入、素材内容和自然语言要求中识别。
- 图片位置占位不提供单独开关，只从用户输入内容和自然语言要求中识别。
- 支持“先保存为待整理记录”的轻量入口；该入口只保存输入和识别结果，不创建文章。

用户目的说明可以很简单，也可以很详细。系统不能要求用户必须使用固定模板，但需要尽量识别以下信息：

- 用户希望产出的内容类型，例如博客草稿、学习笔记、思考随笔、排错复盘或资料整理。
- 用户希望处理原文的方式，例如规范整理、补齐结构、润色表达、保留原意、扩写大纲或压缩摘要。
- 用户明确要求保留的内容，例如代码块、报错信息、链接、图片位置、引用来源和个人观点。
- 用户不希望 AI 做的事情，例如不要虚构经历、不要改动结论、不要生成太长、不要自动发布。

如果用户从其他地方复制了一份笔记，并说明“规范整理、保留图片位置占位、写入新建文章”，系统需要把这类说明识别为本次任务目标，而不是把它当作正文内容直接写进文章。

用户点击“生成草稿方案”后进入分析流程。

生成前必须校验：

- 输入内容不能为空。
- 必须已经配置可用 AI 模型。
- 如果输入少于 10 个中文字符或只有 1 个关键词，仍允许生成，但必须进入“低信息量草稿”策略。

如果校验失败，不发起 AI 请求。

### 4.2 Step 2：识别用户目的并求证

系统在正式生成草稿方案前，必须先识别用户本次目的，并判断是否需要向用户求证。

目的识别需要输出：

- 用户想做什么。
- 用户提供了什么材料。
- 用户希望生成哪类文章或笔记。
- 用户希望保留哪些原始信息。
- 用户是否在输入内容或自然语言要求中表达了保留图片位置占位。
- 用户是否已经表达了分类、二级分类、标题或写作风格偏好。
- 当前输入中有哪些不明确或互相冲突的地方。

如果用户表达清楚，系统可以继续生成草稿方案，但仍必须在预览页展示 AI 对用户目的的理解，供用户人工审核。

如果用户表达不清楚，系统必须进入求证流程。求证问题要求：

- 每次最多展示 3 个关键问题，避免打断随手记录体验。
- 优先询问会影响文章创建的事项，例如一级分类、二级分类、写作目标、用户要求保留图片占位但位置无法判断、是否允许扩写。
- 不询问无关细节，不把用户逼成填写复杂表单。
- 用户可以回答问题后重新生成，也可以选择“按当前理解继续生成预览”。

需要求证的典型情况：

- 用户只粘贴了笔记，没有说明要整理成哪种内容。
- 用户同时提到多个目标，例如既要教程又要随笔，且两者结构冲突。
- 用户要求保留图片位置，但原文中图片位置无法识别。
- 用户输入跨多个主题，且无法判断主文章主题。
- 一级分类或二级分类判断置信度较低。
- 用户要求“写入新建文章”，但标题、分类或整理方式缺少可判断依据。

求证流程的人工审核要求：

- AI 可以给出默认理解，但必须说明这是推断。
- 用户未确认前，系统不得创建文章文件。
- 用户回答求证问题后，系统需要把回答合并进上下文，并重新生成或修正草稿方案。
- 如果用户选择“按当前理解继续”，该选择必须写入任务历史，后续预览仍需显示低置信度或风险提示。

### 4.3 Step 3：收集上下文

系统在生成草稿方案前，需要收集以下上下文：

- 可用一级分类注册表。
- 每个一级分类对应的 `prefix1`。
- 每个一级分类对应的目录。
- 每个一级分类对应的中文名称。
- 每个一级分类对应的封面图。
- 已有文章列表。
- 从已有文章文件名中提取出的二级前缀 `prefix2`。
- AI 从用户输入和素材中识别出的写作目标。
- AI 识别出的用户目的。
- 用户对求证问题的回答。
- 用户在输入或素材中表达的目标读者、期望篇幅、写作风格等偏好。
- 最近生成但未创建的 Agent 方案，避免重复标题和重复主题。

已有文章文件名格式：

```text
prefix1-prefix2-title.md
```

示例：

```text
ts-ai-AI工具使用经验.md
ge-blog-博客架构规划.md
```

从上述文件中提取：

- `prefix1`: `ts`
- `prefix2`: `ai`
- `title`: `AI工具使用经验`

### 4.4 Step 4：AI 生成结构化草稿方案

系统向 AI 发送用户输入和上下文，要求 AI 只返回结构化 JSON。

返回结构必须包含：

```ts
interface AIDraftPlan {
  schemaVersion: string
  approvalMode: 'request-approval' | 'delegate-approval' | 'full-access'
  writingGoal: 'auto' | 'blog-draft' | 'study-note' | 'pitfall-review' | 'project-log' | 'thought-essay' | 'resource-digest' | 'outline-expansion'
  userIntent: string
  inputType: string
  clarificationRequired: boolean
  clarificationQuestions: string[]
  primarySlug: string
  primaryName: string
  prefix1: string
  prefix2: string
  title: string
  tags: string[]
  description: string
  outline: string[]
  imagePlaceholders: string[]
  bodyMarkdown: string
  missingInfoQuestions: string[]
  riskFlags: string[]
  confidence: number
  reviewChecklist: string[]
  rationale: string
}
```

字段要求：

- `schemaVersion`：结构版本，便于后续兼容，初始值建议为 `1.0`。
- `approvalMode`：用户发送前选择的权限模式。
- `writingGoal`：本次写作目标。
- `userIntent`：AI 对用户本次目的的理解，例如“将复制来的学习笔记规范整理为博客草稿，并保留图片位置占位”。
- `inputType`：AI 判断的输入类型，例如 `灵感速记`、`学习笔记`、`踩坑复盘`。
- `clarificationRequired`：是否需要用户进一步求证。
- `clarificationQuestions`：需要用户回答的关键问题，最多 3 个。
- `primarySlug`：一级分类英文目录名，例如 `tech-study`。
- `primaryName`：一级分类中文名，例如 `技术研习`。
- `prefix1`：一级前缀，例如 `ts`。
- `prefix2`：二级前缀，例如 `ai`、`blog`、`docker`。
- `title`：文章标题，不包含文件前缀和 `.md`。
- `tags`：文章标签，建议 2 到 5 个。
- `description`：文章简介，建议 100 到 200 字。
- `outline`：正文结构大纲，至少包含 3 个小节标题；低信息量输入可少于 3 个，但必须说明待补充。
- `imagePlaceholders`：从原始输入中识别出的图片位置占位说明，按出现顺序记录。
- `bodyMarkdown`：标准 Markdown 正文，不包含 Front Matter。
- `missingInfoQuestions`：为了继续完善文章，建议用户补充的问题。
- `riskFlags`：潜在风险，例如 `信息不足`、`事实待验证`、`分类不确定`、`可能包含敏感信息`。
- `confidence`：AI 对分类和命名判断的置信度，范围 0 到 1。
- `reviewChecklist`：创建前建议用户重点审核的项目，例如分类、标题、图片占位、事实依据。
- `rationale`：分类、标题和正文组织的简要理由。

正文生成质量要求：

- 必须保留用户原始输入中的关键事实、观点、步骤、代码和错误信息。
- 可以调整顺序和表达，但不能改变用户原意。
- 可以补充连接句、标题层级、总结和待补充问题。
- 信息不足时，不得硬写成长文；应生成短草稿和待补充清单。
- 对链接、引用、外部资料只能做“待阅读/待验证”标记，不能编造其内容。
- 对图片、截图、附件等非文本素材，必须保留位置占位，不能编造图片内容。
- 对技术步骤必须避免虚构命令、参数、版本号和效果。
- 对个人经历必须避免虚构结果、情绪、事件和时间线。

图片占位处理要求：

- 如果原文包含 Markdown 图片语法，例如 `![说明](图片路径)`，正文中应尽量保留该位置。
- 如果原文包含 HTML 图片标签、文件名、截图说明或“此处有图”等提示，应转换为明确占位，例如 `【图片占位：原笔记第 2 张图，待上传】`。
- 如果用户明确要求保留图片位置，占位必须出现在正文中对应语义位置，不能全部集中到文章末尾。
- 如果无法判断图片位置，必须在 `clarificationQuestions` 或 `riskFlags` 中提示用户确认。
- `imagePlaceholders` 只记录占位信息，不代表图片已经上传或保存。

### 4.5 Step 5：草稿方案校验

AI 返回后，前端必须校验结果。

校验规则：

- `schemaVersion` 必须存在。
- `approvalMode` 必须在允许枚举范围内。
- `writingGoal` 必须在允许枚举范围内。
- `userIntent` 不能为空。
- `primarySlug` 必须存在于一级分类注册表。
- `primaryName` 必须与注册表中的中文名一致。
- `prefix1` 必须与注册表中的一级前缀一致。
- `prefix2` 不能为空。
- `prefix2` 不能包含连字符 `-`。
- `title` 不能为空。
- `description` 不能为空。
- `bodyMarkdown` 不能为空。
- `bodyMarkdown` 不能包含 Front Matter 分隔符 `---`。
- `confidence` 必须在 0 到 1 之间。
- `clarificationRequired` 必须是布尔值。
- `clarificationQuestions` 必须是数组，最多 3 个。
- `imagePlaceholders` 必须是数组。
- `missingInfoQuestions` 必须是数组。
- `riskFlags` 必须是数组。
- `reviewChecklist` 必须是数组。

校验失败时，系统不允许创建文章。

用户可以选择：

- 重新生成。
- 手动修改方案中的可编辑字段。
- 取消本次生成。

### 4.6 Step 6：展示草稿预览

校验通过后，页面进入预览确认状态。

预览区域必须展示：

- 一级分类。
- 一级前缀。
- 二级前缀。
- 文章标题。
- 生成后的文件名。
- 生成后的文章路径。
- 标签。
- 文章简介。
- 写作目标。
- 用户目的理解。
- 输入类型。
- 正文大纲。
- 求证问题。
- 图片位置占位。
- 待补充问题。
- 风险提示。
- 置信度。
- 创建前审核清单。
- AI 判断理由。
- Markdown 正文预览。
- 本次权限模式，以及权限模式对后续创建或写入的影响。

生成文件名规则：

```text
prefix1-prefix2-title.md
```

生成路径规则：

```text
source/_posts/{primarySlug}/{prefix1}-{prefix2}-{title}.md
```

示例：

```text
source/_posts/tech-study/ts-ai-AI写作助手工作流设计.md
```

### 4.7 Step 7：用户确认、修改或授权执行方案

创建文章前，系统必须根据本次权限模式决定是否需要用户手动确认。

预览页中允许用户修改：

- 用户目的理解。
- 一级分类。
- 二级前缀。
- 文章标题。
- 标签。
- 文章简介。
- 写作目标。
- 正文大纲。
- 图片占位文本。
- 正文 Markdown。

用户修改一级分类后，系统必须同步更新：

- `primarySlug`
- `primaryName`
- `prefix1`
- 默认封面图
- 生成文件名
- 生成路径

用户修改二级前缀后，系统必须重新校验：

- 不能为空。
- 不能包含 `-`。
- 建议使用英文、数字或下划线。

只有全部校验通过，才允许进入创建草稿流程。

不同权限下的创建规则：

- 请求批准：必须由用户点击“创建草稿”。
- 替我审批：如果 `confidence >= 0.6`、`clarificationRequired` 为 `false`、`riskFlags` 不包含高风险项，系统可以自动创建草稿；否则降级为请求批准。
- 完全访问权限：如果 schema 校验和业务校验全部通过，系统可以自动创建草稿并写入正文；但遇到高风险、低置信度、求证未完成或图片占位不明确时，必须暂停并请求用户确认。

如果 `confidence` 低于 0.6，或 `riskFlags` 包含 `分类不确定`、`事实待验证`、`信息不足`，页面必须提示用户先检查方案；仍允许用户确认创建，但确认按钮需要明确显示“确认并创建草稿”。

如果 `clarificationRequired` 为 `true`，页面必须展示求证问题，并默认禁用“创建草稿”按钮。用户需要先完成以下任一操作：

- 回答求证问题并重新生成方案。
- 手动修改方案，使分类、标题、正文和图片占位达到可创建状态。
- 勾选“我已知晓不确定项，按当前理解创建草稿”。

人工审核必须覆盖以下内容：

- AI 对用户目的的理解是否正确。
- 一级分类是否符合文章性质。
- 二级前缀是否复用或新建合理。
- 标题是否符合命名规则且能概括内容。
- 正文是否保留了原始笔记中的关键事实、观点、代码和报错信息。
- 图片位置占位是否与原笔记顺序和语义位置一致。
- 待补充问题和风险提示是否可以接受。

如果正文包含“待补充”标记，页面必须提示：

```text
这是一篇可继续完善的草稿，AI 没有补全缺失事实。
```

### 4.8 Step 8：创建草稿文件

用户点击“创建草稿”后，系统调用现有创建文章接口。

调用参数：

```ts
{
  title: string
  prefix1: string
  prefix2: string
}
```

创建接口只负责生成符合项目规范的文章文件和基础 Front Matter。

创建服务必须遵守与以下命令等价的参数规范：

```bash
npx hexo np <一级前缀> <二级前缀> "标题"
```

不论底层目标是本地文件系统还是远程仓库，前端都只调用统一创建接口，不在页面中区分本地模式和在线模式。

如果创建失败，系统必须展示失败原因，并保留当前预览内容，用户可修改后重试。

### 4.9 Step 9：写入 AI 生成正文

文章创建成功后，系统读取创建接口返回的文章路径，并调用文章更新接口写入完整内容。

写入内容由 Front Matter 和正文组成。

Front Matter 字段必须包含：

```yaml
title: 文章标题
date: 创建时间
updated:

categories:
  - [一级分类中文名, 二级前缀]

tags:
  - 标签1
  - 标签2

description: 文章简介

layout: post
comments: true
permalink:
excerpt:
published: true
lang: zh-CN

cover: /img/covers/对应一级分类封面.svg
sticky:

slug:
status: draft

series:
series_order:
```

正文使用 AI 生成并经用户确认或授权后的 `bodyMarkdown`。

### 4.10 Step 10：创建成功反馈

写入成功后，系统需要：

- 提示“草稿创建成功”。
- 刷新文章列表缓存。
- 将任务历史标记为已创建。
- 跳转到编辑器页面。

跳转地址格式：

```text
/editor?file={文章相对路径}
```

示例：

```text
/editor?file=tech-study%2Fts-ai-AI写作助手工作流设计.md
```

### 4.11 Step 11：任务历史

Agent 模式页面左侧显示任务历史。

历史记录字段：

```ts
interface AIWritingSession {
  id: string
  mode: 'agent'
  approvalMode: string
  writingGoal: string
  userIntent: string
  inputType: string
  inputSummary: string
  title: string
  primaryName: string
  prefix2: string
  path?: string
  status: 'planned' | 'created' | 'failed'
  confidence: number
  riskFlags: string[]
  clarificationRequired: boolean
  imagePlaceholderCount: number
  createdAt: string
}
```

历史记录用途：

- 查看最近生成的草稿方案。
- 快速跳转到已创建文章。
- 区分已创建和未创建方案。
- 继续编辑未创建方案。
- 复用历史输入重新生成。
- 避免重复标题和重复主题。
- 查看当时用户目的、求证状态和图片占位数量。

历史记录保存到浏览器 localStorage。

### 4.12 Agent 稳定性要求

Agent 模式必须以“稳定生成、稳定校验、稳定写入”为设计目标。

状态稳定性要求：

- 每次生成都要有唯一 `sessionId`。
- 每次创建文章都要有幂等标识，避免用户重复点击导致重复创建。
- 每次发送都要记录本次 `approvalMode`，自动执行动作必须可追踪。
- 每个 LangGraph 节点只能做一类事情，不能在生成节点里直接写文件。
- 求证节点只负责识别不确定项和生成问题，不能替用户回答问题。
- `plan_draft` 节点只负责生成方案。
- `validate_plan` 节点只负责校验方案。
- `prepare_preview` 节点只负责预览。
- `create_post` 和 `write_markdown` 必须在权限允许且校验通过后才执行；请求批准模式下必须等待用户确认。
- 节点失败时必须返回可恢复状态，而不是丢失用户输入。

输出稳定性要求：

- AI 返回必须是严格 JSON。
- JSON 解析失败时最多自动重试一次，重试提示词必须要求修复格式，不重新发挥内容。
- 结构校验失败时不进入创建流程。
- 低置信度不阻断预览，但必须显著提示。
- 同一输入在短时间内重复生成，应尽量保持分类和标题稳定，除非用户修改写作目标。

写入稳定性要求：

- 创建文章前不写入任何文章文件。
- 创建成功但写入正文失败时，必须保留已创建路径并提示用户进入编辑器手动恢复。
- 写入正文时必须保留合法 Front Matter。
- 写入失败不能清空任务历史和预览内容。

用户体验稳定性要求：

- 生成中显示明确进度。
- 长文本输入时不得卡死页面。
- 用户离开页面前，如果存在未创建方案，需要提示是否保留。
- 历史记录损坏时不能影响主流程。

## 5. LLM 模式需求

LLM 模式用于“基于当前文章对话式修改内容”。

它主要出现在编辑器中，也可以在 AI 辅助写作页面中作为说明入口，引导用户打开文章后使用。

### 5.1 Step 1：打开文章

用户在文章列表、仪表盘或 Agent 创建成功后进入编辑器。

编辑器加载当前文章完整 Markdown 内容，包括 Front Matter 和正文。

AI 侧边栏接收：

- 当前文章完整内容。
- 当前正文内容。
- 当前标题。
- 当前 Front Matter。
- 当前选中文本。
- 当前光标位置。

### 5.2 Step 2：展开 AI 侧边栏

用户点击 AI 助手按钮后，侧边栏展开。

侧边栏包含：

- 对话历史。
- 模型选择。
- 发送前权限选择。
- 当前作用范围提示。
- 快捷操作。
- 指令输入框。
- 生成结果预览。
- 采纳/放弃按钮。

侧边栏可以收起。收起后不丢失当前对话历史。

### 5.3 Step 3：识别作用范围

系统根据编辑器状态判断本次 AI 操作作用范围。

如果用户选中文本：

```text
作用范围：选中文本
```

如果没有选中文本但有光标位置：

```text
作用范围：光标位置
```

如果用户选择全文操作：

```text
作用范围：整篇文章
```

系统不能在用户不知情的情况下对全文执行大范围修改。

### 5.4 Step 4：用户输入修改指令

用户可以输入自然语言指令，例如：

```text
把这一段写得更自然一点
```

```text
删除重复表达，只保留关键观点
```

```text
在这里续写一个注意事项小节
```

```text
把全文整理成更清晰的技术文章结构
```

用户发送指令前可以选择权限模式，默认“请求批准”。LLM 模式必须根据权限模式决定 AI 结果是否只展示预览，还是在校验通过后自动采纳。

### 5.5 Step 5：快捷操作

LLM 侧边栏提供快捷操作。

快捷操作包括：

- 优化润色。
- 续写内容。
- 改写风格。
- 总结摘要。
- 生成大纲。
- 生成小节。
- 优化标题。
- 优化描述。
- 翻译为英文。
- 翻译为中文。

有选中文本时，快捷操作默认作用于选中文本。

无选中文本时，快捷操作根据类型决定：

- 续写：插入到光标位置。
- 生成大纲：插入到光标位置。
- 总结摘要：生成预览，不直接覆盖正文。
- 优化标题：只建议标题，不直接修改正文。
- 优化描述：只建议 description，不直接修改正文。

### 5.6 Step 6：AI 生成修改方案

LLM 生成结果是否可以直接写入文章，取决于本次权限模式。

请求批准模式下，系统必须先展示预览，不能直接写入。

预览内容根据操作类型展示：

- 插入：展示将插入的内容。
- 替换：展示原内容和新内容。
- 删除：展示将删除的内容摘要。
- 全文重写：展示新全文预览。
- Front Matter 修改：展示字段名、旧值、新值。

### 5.7 Step 7：用户采纳或放弃

请求批准模式下，用户可以选择：

- 采纳。
- 放弃。
- 继续追问。
- 重新生成。

请求批准模式下，只有用户点击“采纳”后，系统才修改编辑器内容。

替我审批模式下：

- 插入到光标位置、替换当前选区、优化标题、优化 description 等低风险操作，在校验通过后可以自动采纳。
- 删除、全文重写、修改分类、修改 `published`、修改 `status`、修改 permalink、批量 Front Matter 修改必须先展示预览并等待用户确认。
- 如果作用范围不明确，必须降级为请求批准。

完全访问权限下：

- 在当前文章范围内，插入、替换、删除正文片段、全文重写和允许的 Front Matter 修改可以在校验通过后自动执行。
- 自动执行前必须生成操作摘要并记录到对话历史。
- 如果操作涉及发布、删除文件、提交 Git、部署、修改主题源码或系统配置，必须拒绝自动执行。

采纳后：

- 更新编辑器内容。
- 触发现有编辑器变更逻辑。
- 保留自动保存机制。
- 在对话历史中记录本次操作。

放弃后：

- 不修改文章。
- 保留对话历史。

### 5.8 Step 8：执行插入操作

插入操作适用于：

- 续写。
- 新增小节。
- 插入示例。
- 插入总结。
- 插入代码块。

执行规则：

- 有光标位置时，插入到光标位置。
- 无法定位光标时，插入到正文末尾。
- 插入内容必须保持 Markdown 格式。
- 插入后光标移动到新增内容末尾。

### 5.9 Step 9：执行替换操作

替换操作适用于：

- 润色选区。
- 改写选区。
- 翻译选区。
- 重构某一段。

执行规则：

- 必须存在选中文本。
- 只替换选中文本范围。
- 不修改选区之外的内容。
- 替换后清空选区。

### 5.10 Step 10：执行删除操作

删除操作适用于：

- 删除重复段落。
- 删除跑题内容。
- 删除用户指定小节。
- 删除空洞表达。

执行规则：

- 删除前必须展示将删除内容的摘要。
- 请求批准和替我审批模式下，删除前必须让用户确认采纳。
- 完全访问权限下，删除正文片段可以在校验通过后自动执行，但必须记录删除摘要。
- 不能直接删除 Front Matter。
- 如果删除范围不明确，AI 必须先解释建议删除哪些内容，由用户确认。

### 5.11 Step 11：执行全文重写

全文重写适用于：

- 将杂乱文章整理为完整技术文章。
- 调整文章结构。
- 将口语化内容改成博客文章。

执行规则：

- 必须明确提示这是全文级修改。
- 必须展示全文预览。
- 必须保留或重新生成合法 Front Matter。
- 请求批准和替我审批模式下，用户采纳后才替换全文。
- 完全访问权限下，全文重写可以在校验通过后自动替换，但必须记录改动摘要，并保留可恢复的旧内容快照。

## 6. 知识库稳定模式需求

知识库稳定模式用于“基于现有文章数据进行可靠问答”。该模式只读取本地博客文章和本地生成的索引，不创建文章、不修改文章、不发布文章，也不执行 Git、部署或删除操作。

该模式的核心目标不是让 LLM 自由发挥，而是让系统先准确找到本地依据，再由 LLM 基于依据组织回答。

### 6.1 设计原则

- 以本地 Markdown 文件和 Front Matter 为最终事实来源。
- 文章清单、数量、分类、标签、状态、路径等问题必须通过本地扫描或本地索引计算，不能让 LLM 猜测。
- 涉及具体文章内容的问题必须返回引用来源，例如文章标题、文件路径、相关小节。
- 找不到依据时必须回答“当前文章中没有找到明确依据”，不能编造。
- 本模式不需要引入数据库。
- 默认不引入独立向量数据库。
- 本地索引必须可删除、可重建，不能成为唯一数据源。

### 6.2 典型问题

知识库稳定模式需要支持以下问题：

- 我一共有多少篇文章？
- 我有哪些文章？
- 技术研习分类下有哪些文章？
- 我写过哪些关于 AI、Jetson、Hexo、Docker 的文章？
- 最近更新了哪些文章？
- 哪些文章还是草稿？
- 某篇文章讲了什么？
- 我有没有写过某个主题？
- 某个问题以前在哪篇文章里记录过？
- 根据已有文章，总结我最近关注的技术方向。

### 6.3 本地文章扫描

系统需要提供稳定的本地文章扫描能力。扫描范围以实际项目结构为准，优先包括：

```text
blog-content/source/_posts/**/*.md
blog-content/source/_drafts/**/*.md
```

扫描时需要解析：

- 文件路径。
- 文件名。
- Front Matter。
- 标题。
- 创建时间。
- 更新时间。
- 一级分类。
- 二级分类。
- 标签。
- `status`。
- `published`。
- `description`。
- 正文纯文本。
- Markdown 标题层级。
- 文章摘要。
- 文件修改时间。
- 内容 hash。

如果 `blog-content/source/_posts/index.md` 与实际文件不一致，知识库稳定模式必须以实际文件扫描结果为准，并在回答中提示“文章索引可能需要同步”。

### 6.4 本地索引文件

为提升问答速度，可以生成本地索引文件，但索引文件不是数据库。

推荐索引文件：

```text
admin-dev/.cache/article-index.json
admin-dev/.cache/article-chunks.json
```

索引要求：

- 可以由本地文章重新生成。
- 文件不存在时自动扫描重建。
- 文章内容 hash 或修改时间变化时自动更新对应索引。
- 索引损坏时自动丢弃并重建。
- 不把索引作为唯一事实来源。

文章索引结构建议：

```ts
interface ArticleIndexItem {
  path: string
  fileName: string
  title: string
  date?: string
  updated?: string
  categories: string[][]
  tags: string[]
  status?: 'draft' | 'wip' | 'published'
  published?: boolean
  description?: string
  headings: string[]
  excerpt: string
  mtime: string
  contentHash: string
}
```

分块索引结构建议：

```ts
interface ArticleChunk {
  id: string
  articlePath: string
  articleTitle: string
  headingPath: string[]
  text: string
  startLine?: number
  endLine?: number
  contentHash: string
}
```

### 6.5 问答流程

知识库稳定模式建议使用 LangChain 或 LangGraph 实现本地检索增强问答。流程如下：

1. `scan_or_load_index`：读取本地索引；索引缺失、过期或损坏时扫描 Markdown 文件重建。
2. `classify_question`：判断问题类型，区分清单统计、元数据查询、具体内容查询、主题总结和无法回答问题。
3. `retrieve_evidence`：根据问题类型检索相关文章、Front Matter 或正文分块。
4. `validate_evidence`：校验证据是否来自本地文章，路径是否存在，内容 hash 是否匹配。
5. `answer_with_citations`：基于证据生成回答，并附带来源。
6. `fallback`：证据不足时说明未找到明确依据，并给出可继续查询的方向。

不同问题类型的处理规则：

- 清单统计类：不调用 LLM 生成事实，直接由扫描结果计算，再让 LLM 只负责组织表达。
- 元数据查询类：基于 Front Matter 和文件路径过滤。
- 具体内容类：先检索相关正文分块，再基于分块回答。
- 主题总结类：必须列出参与总结的文章来源。
- 模糊问题：先给出最相关结果和不确定说明，必要时询问用户是否缩小范围。

### 6.6 “我有哪些文章”回答要求

当用户询问“我有哪些文章”“我一共有哪些文章”“我写过哪些文章”时，系统必须：

- 扫描或读取最新本地文章索引。
- 返回文章总数。
- 按一级分类分组展示。
- 展示文章标题、二级分类、状态和相对路径。
- 如果文章数量较多，默认展示完整统计和前若干篇，并提供“展开全部”或“导出列表”入口。
- 不使用 LLM 记忆回答文章清单。
- 不根据旧对话历史猜测文章数量。

回答格式示例：

```text
当前共扫描到 36 篇文章。

技术研习（18 篇）
- Jetson Ollama 部署 LLM｜状态：published｜tech-study/ts-jetson-Ollama部署LLM.md
- Claude Code 入门与安装｜状态：draft｜tech-study/ts-claude-code入门与安装.md

成长随笔（8 篇）
- 博客写作完整指南｜状态：draft｜growth-essay/ge-blog-博客写作完整指南.md
```

### 6.7 具体内容问答要求

当用户询问文章具体内容时，系统必须先检索本地文章内容，再回答。

回答必须包含：

- 直接答案。
- 引用来源。
- 相关小节或路径。
- 如有不确定，说明不确定原因。

引用来源格式建议：

```text
来源：
- 《文章标题》：blog-content/source/_posts/tech-study/ts-ai-agent入门笔记.md
- 小节：## Agent 的基本概念
```

禁止行为：

- 不得把未检索到的内容当成文章已有内容。
- 不得把模型常识伪装成用户文章内容。
- 不得引用不存在的文章路径。
- 不得因为某个主题看起来相关就声称用户写过。

### 6.8 是否需要向量数据库

本系统初始版本不需要引入向量数据库。

原因：

- 系统主要是本地博客文件管理，文章数量规模通常不大。
- “我有哪些文章”“有哪些分类”“某类文章数量”这类问题需要精确扫描，不适合依赖向量检索。
- 向量数据库会引入额外服务、数据同步、索引一致性和部署复杂度，不符合“本地操作、不涉及数据库”的定位。
- 稳定问答更依赖可验证来源、文件路径、Front Matter 和本地索引，而不是仅靠语义相似度。

推荐方案：

- 第一阶段使用本地文件扫描 + Front Matter 解析 + 标题/标签/正文关键词检索 + Markdown 分块。
- 如需更好的语义召回，可以增加本地 embedding 缓存文件，但不引入独立向量数据库服务。
- embedding 缓存必须可重建，并与文章 hash 绑定。
- 即使使用 embedding，也必须把召回结果回到原文分块校验后再回答。

只有在文章数量显著增长、关键词检索无法满足语义查询、并且用户接受本地索引复杂度增加时，才考虑引入轻量本地向量索引。即便如此，也应优先选择文件型、本地型、可重建方案，而不是远程数据库服务。

### 6.9 知识库稳定性要求

- 回答前必须确认索引与本地文件一致。
- 文件扫描失败时必须提示失败原因。
- 单篇文章解析失败不能导致整个知识库不可用，应记录失败文章并继续处理其他文章。
- 回答必须标记数据来源时间或扫描时间。
- 用户要求“准确回答”时，应优先输出可验证清单和引用，而不是长篇生成式总结。
- 本模式下默认不使用发送前写入权限，因为它是只读问答模式。

## 9. 接口需求

### 9.1 AI 编排技术要求

AI 辅助写作后端必须使用 LangChain 或 LangGraph 实现模型调用与工作流编排。

Agent 模式优先使用 LangGraph，因为该模式是固定工作流，包含明确的状态和节点流转。推荐状态结构：

```ts
interface AgentWritingState {
  sessionId: string
  userInput: string
  approvalMode: 'request-approval' | 'delegate-approval' | 'full-access'
  writingGoal: string
  userIntent?: string
  userPreferences?: Record<string, unknown>
  clarificationRequired: boolean
  clarificationQuestions: string[]
  clarificationAnswers: string[]
  categoryRegistry: CategoryRegistryItem[]
  existingPrefixes: Record<string, string[]>
  draftPlan?: AIDraftPlan
  imagePlaceholders: string[]
  validationErrors: string[]
  riskFlags: string[]
  reviewed: boolean
  confirmed: boolean
  createdPath?: string
  finalMarkdown?: string
}
```

Agent 模式 LangGraph 节点要求：

1. `collect_context`：收集分类注册表、已有文章和已有二级前缀。
2. `parse_user_intent`：识别用户目的、输入材料类型、明确要求和保留项。
3. `clarify_intent`：判断是否需要求证；需要时生成最多 3 个关键问题，并等待用户回答或确认按当前理解继续。
4. `classify_input`：识别输入类型和写作目标，必要时优先采用用户在自然语言中明确表达的目标。
5. `plan_draft`：调用 LLM 生成结构化 `AIDraftPlan`。
6. `validate_plan`：校验一级分类、二级前缀、标题、描述、正文、图片占位和风险标记。
7. `prepare_preview`：生成预览数据和人工审核清单，不写入文章。
8. `wait_user_confirm`：等待用户确认或修改方案。
9. `create_post`：调用统一文章创建接口。
10. `write_markdown`：调用统一文章更新接口写入正文。
11. `finish`：返回创建结果和文章路径。

LLM 模式可以使用 LangChain 消息链实现对话和编辑建议生成。对于替换、插入、删除、全文重写等操作，AI 返回结果必须结构化，不能只返回自由文本。

推荐编辑操作结构：

```ts
interface AIEditOperation {
  type: 'insert' | 'replace' | 'delete' | 'rewrite' | 'frontmatter'
  scope: 'selection' | 'cursor' | 'document' | 'frontmatter'
  approvalMode: 'request-approval' | 'delegate-approval' | 'full-access'
  requiresManualApproval: boolean
  summary: string
  oldText?: string
  newText?: string
  frontMatterPatch?: Record<string, unknown>
}
```

知识库稳定模式可以使用 LangChain 或 LangGraph 实现本地检索增强问答。推荐状态结构：

```ts
interface KnowledgeQAState {
  sessionId: string
  question: string
  questionType: 'inventory' | 'metadata' | 'content' | 'summary' | 'unknown'
  indexStatus: 'fresh' | 'stale' | 'rebuilt' | 'failed'
  scannedAt: string
  articles: ArticleIndexItem[]
  retrievedChunks: ArticleChunk[]
  answer: string
  citations: KnowledgeCitation[]
  confidence: number
  warnings: string[]
}

interface KnowledgeCitation {
  title: string
  path: string
  headingPath?: string[]
  lines?: [number, number]
}
```

### 9.2 提示词工程要求

Agent 模式、LLM 模式和知识库稳定模式都必须有明确的系统提示词、任务提示词、输出格式约束和后置校验。提示词不能只依赖“请生成一篇文章”这类宽泛描述，必须把边界、可用上下文、禁止行为和失败处理写清楚。

#### 9.2.1 通用防幻觉规则

所有 AI 提示词都必须包含以下约束：

- 只能使用用户输入、文章上下文、分类注册表、已有文章列表和系统明确提供的规则。
- 不得编造不存在的一级分类、二级前缀、文件路径、文章路径、接口能力、构建命令或项目配置。
- 必须区分“用户目的说明”和“文章正文素材”，不能把用户对任务的要求原样写进正文。
- 不确定时必须返回低置信度，并在 `rationale` 或 `summary` 中说明不确定原因。
- 不得声称已经创建、保存、发布、删除或修改文件；AI 只能生成方案或编辑操作，真实写入由系统接口完成。
- 不得伪造或提升用户选择的权限模式。
- 请求批准模式下不得跳过用户确认步骤。
- 替我审批和完全访问权限下，只能在权限边界、schema 校验和业务校验全部通过时自动执行。
- 遇到高风险、低置信度、求证未完成、图片占位不明确或作用范围不明确时，必须降级为请求批准。
- 不得输出与要求 schema 无关的解释文本。
- 不得生成 Front Matter 中未被项目规范允许的字段，除非用户明确要求且系统允许。
- 不得将外部事实写成确定事实；如果用户输入中没有提供依据，只能写成建议、经验或待验证表述。
- 对技术内容要避免虚构 API、版本号、错误码、命令参数和性能数据。
- 对个人经历、项目经历、面试经历等内容，不能编造用户没有提供的事实。

#### 9.2.2 Agent 模式提示词要求

Agent 模式的 `plan_draft` 节点提示词必须明确说明：

- 输入内容可能是杂乱笔记，不要求用户提供标准 Markdown。
- 输入可能只是随手记录的一句话，不能因为内容少就编造细节。
- 用户可能会用自然语言说明目的，例如“整理这份笔记”“保留图片占位”“写入新文章”；这些说明应作为任务约束，不应直接进入正文。
- 必须先识别用户目的，如果目的影响分类、结构、扩写程度或图片占位，但无法判断清楚，必须设置 `clarificationRequired: true`。
- 求证问题必须具体、少量、可回答，不能把所有可选项都抛给用户。
- 必须先判断输入类型和写作目标，再决定正文结构。
- 一级分类只能从分类注册表中选择。
- `prefix1` 必须来自分类注册表，不能自行生成。
- `prefix2` 优先复用已有二级前缀；没有合适前缀时才建议新前缀。
- 新 `prefix2` 必须短小、语义清楚，且不能包含 `-`。
- 标题必须是中文自然标题，不包含 `prefix1`、`prefix2` 和 `.md`。
- `bodyMarkdown` 只能包含正文 Markdown，不能包含 Front Matter。
- 如果用户要求保留图片位置，必须在原文对应位置插入清晰的图片占位。
- `description` 必须概括文章主题和读者收益，不能编造未出现的内容。
- 如果用户输入信息不足，只能生成“可继续完善的草稿”，不能伪造细节填满文章。
- 如果分类判断不确定，必须降低 `confidence`，并在 `rationale` 中说明可选分类。
- 必须输出 `missingInfoQuestions`，帮助用户后续补充材料。
- 必须输出 `riskFlags`，标记信息不足、事实待验证、分类不确定等风险。
- 必须输出 `reviewChecklist`，提示用户人工审核分类、二级前缀、标题、正文事实和图片占位。
- 必须根据写作目标选择合适文章结构，例如学习笔记、踩坑复盘、项目日志、思考随笔。

Agent 模式必须要求 AI 输出严格 JSON，并与 `AIDraftPlan` schema 对齐。示例要求：

```text
你只能输出一个 JSON 对象，不要输出 Markdown 代码块，不要输出解释文字。
如果信息不足，请在 bodyMarkdown 中保留“待补充”提示，但不要编造事实。
```

Agent 模式边界情况处理：

- 输入过短：生成简短草稿方案，并提示需要补充背景、目标、关键步骤。
- 用户目的不清晰：先返回求证问题，不能假定用户一定要写成长篇技术博客。
- 用户明确要求规范整理复制来的笔记：以保留原意和结构化整理为主，不能大幅新增未提供内容。
- 用户要求保留图片位置：在正文中保留 `【图片占位：...】`，并在预览中提醒用户后续上传或替换图片。
- 输入跨多个主题：选择最主要主题；在 `rationale` 中说明被拆分的其他主题，不能自动创建多篇文章。
- 输入包含敏感信息：正文中应提醒用户检查敏感信息，不主动扩散或强化敏感内容。
- 输入疑似已有文章改写：生成草稿时避免声称为全新原创经验，只按用户提供内容整理。
- 输入只是资料链接：不能编造链接内容，只能生成待阅读清单、理解框架和待验证问题。
- 输入包含报错信息：保留原始报错，不能改写成不存在的错误码或错误原因。
- 输入包含代码：保留代码语义，不随意替换 API 或参数。
- 输入是个人思考：保留观点的不确定性，不能替用户下绝对结论。
- 无合适分类：选择最接近的注册分类并降低置信度，不能创建新一级分类。
- 无合适二级前缀：建议新 `prefix2`，但必须通过系统校验后才能创建。

#### 9.2.3 LLM 模式提示词要求

LLM 模式提示词必须明确当前作用范围：

- `selection`：只允许修改选中文本。
- `cursor`：只允许生成插入内容。
- `document`：允许全文级建议，但必须在用户确认后才能替换。
- `frontmatter`：只允许修改指定 Front Matter 字段。

LLM 模式必须要求 AI 返回结构化 `AIEditOperation`，并遵守以下规则：

- 替换操作必须包含 `oldText` 和 `newText`。
- 插入操作必须包含 `newText`。
- 删除操作必须包含 `oldText` 和 `summary`。
- 全文重写必须包含完整 `newText`，且 `summary` 必须说明改动范围。
- Front Matter 修改必须包含 `frontMatterPatch`，不能把正文内容混入 Front Matter。
- `requiresManualApproval` 必须根据权限模式、操作类型、作用范围和风险判断生成。
- 如果用户指令不明确，返回澄清型 `summary`，不生成破坏性操作。
- 如果用户要求删除但范围不明确，不能直接删除，必须先建议删除范围。

LLM 模式边界情况处理：

- 用户要求“优化一下”但未选中文本：默认生成全文修改建议预览，不直接改全文。
- 用户要求“删掉废话”但未指定范围：返回建议删除范围，不直接删除。
- 用户要求“写得更真实”：只能基于已有事实增强表达，不能新增经历、数据或结果。
- 用户要求“补充案例”：如果用户没有提供案例，只能插入“示例占位”或提示用户提供案例。
- 用户要求修改标题或描述：只返回 Front Matter patch，不改正文。
- 用户要求大范围结构调整：必须返回全文预览和改动摘要。
- 用户要求执行发布、提交、删除文件等操作：AI 必须拒绝生成执行操作，只能说明该能力需要用户在管理端显式触发。
- 用户选择完全访问权限：只能在当前文章内容编辑范围内扩大自动执行能力，不能理解为系统级完全控制。

#### 9.2.4 知识库稳定模式提示词要求

知识库稳定模式提示词必须明确说明：

- AI 只能基于系统提供的本地文章索引、Front Matter、文章正文分块和引用来源回答。
- 不得使用模型记忆猜测用户有哪些文章。
- 不得把通用知识当作用户文章中的内容。
- 对“我有哪些文章”“我一共有多少文章”等清单统计问题，必须使用系统扫描结果。
- 对具体内容问题，必须先引用相关文章或分块，再组织回答。
- 找不到依据时必须明确说明未找到，不得编造。
- 回答中必须包含来源列表，除非问题是纯粹的索引扫描状态问题。
- 如果索引过期或扫描失败，必须提示用户重新扫描，不能继续生成确定答案。

知识库稳定模式边界情况处理：

- 用户问文章清单：直接使用文章索引，不调用 LLM 生成事实。
- 用户问某个主题是否写过：返回匹配文章和匹配依据；没有匹配时说明未找到。
- 用户问具体方案或经验：只根据召回文章回答，并列出小节来源。
- 用户问“帮我总结所有 AI 相关文章”：先列出参与总结的文章，再总结。
- 用户问外部事实：如果本地文章没有记录，应说明本地知识库中没有依据，可提示用户切换普通对话或自行补充资料。

#### 9.2.5 后置校验要求

AI 输出后，系统必须做后置校验，不能只相信提示词。

Agent 模式校验：

- JSON 可解析。
- 字段完整。
- `schemaVersion` 存在。
- `writingGoal` 在允许枚举范围内。
- `userIntent` 非空。
- `clarificationRequired` 是布尔值。
- `clarificationQuestions` 是数组，且不超过 3 个。
- `primarySlug`、`primaryName`、`prefix1` 与分类注册表一致。
- `prefix2` 合法。
- `bodyMarkdown` 不含 Front Matter。
- `description` 长度和内容符合要求。
- `imagePlaceholders` 是数组。
- `missingInfoQuestions` 是数组。
- `riskFlags` 是数组。
- `reviewChecklist` 是数组。
- `confidence` 在 0 到 1 之间。

LLM 模式校验：

- JSON 可解析。
- `type` 和 `scope` 在枚举范围内。
- `approvalMode` 在允许枚举范围内。
- `requiresManualApproval` 是布尔值。
- 非空字段满足对应操作要求。
- `replace` 的 `oldText` 必须能在当前作用范围内找到。
- `delete` 的 `oldText` 必须能在当前作用范围内找到。
- `frontmatter` patch 只能修改允许字段。
- `rewrite` 不得丢失必要 Front Matter。

知识库稳定模式校验：

- 回答依据必须来自本地文章索引或本地正文分块。
- 引用路径必须存在。
- 引用文章标题必须与 Front Matter 或文件解析结果一致。
- 清单统计类回答的数量必须与扫描结果一致。
- 回答中的文章标题不得超出检索结果范围。
- 具体内容回答必须至少包含一个引用来源。
- 找不到依据时不能输出确定性结论。

校验失败时，系统必须阻止写入，并提示用户重新生成或手动调整。

## 10. 异常处理

### 10.1 未配置 AI 模型

情况：

用户点击生成时没有可用模型。

处理：

- 提示用户先配置 AI 模型。
- 提供跳转设置页入口。
- 不发起生成请求。

### 10.2 AI 返回非 JSON

情况：

Agent 模式要求 JSON，但 AI 返回了普通文本。

处理：

- 显示“AI 返回格式异常，请重新生成”。
- 保留用户输入。
- 不创建文章。

### 10.3 分类不可用

情况：

AI 返回的一级分类不在注册表中。

处理：

- 标记方案无效。
- 提示用户重新生成或手动选择一级分类。
- 不创建文章。

### 10.4 AI 输出疑似幻觉

情况：

AI 输出了不存在的分类、路径、接口能力、命令、项目配置、外部事实、用户经历、技术参数或无法从上下文验证的确定性结论。

处理：

- 标记输出为高风险。
- 阻止自动写入。
- 提示用户检查并重新生成。
- 如果只是文章正文中的可疑事实，应在预览中高亮或提示“该内容缺少输入依据”。

### 10.5 用户指令超出边界

情况：

用户要求 AI 发布文章、提交 Git、删除文件、修改主题源码、绕过确认、创建未注册一级分类，或执行其他超出 AI 写作助手边界的操作。

处理：

- AI 不生成执行操作。
- 页面提示该操作不属于 AI 写作助手自动执行范围。
- 如果管理端已有对应显式按钮或流程，引导用户使用正式流程。
- 即使用户选择“完全访问权限”，也不能放行上述操作。

### 10.6 二级前缀非法

情况：

`prefix2` 为空或包含 `-`。

处理：

- 输入框显示校验错误。
- 禁用“创建草稿”按钮。

### 10.7 创建文章失败

情况：

统一创建服务执行失败，或目标文件已存在。

处理：

- 展示后端返回错误。
- 保留草稿方案。
- 允许用户修改标题或前缀后重试。

### 10.8 写入正文失败

情况：

文章创建成功，但更新文章内容失败。

处理：

- 显示保存失败。
- 提供跳转到已创建文章的入口。
- 提醒用户可手动编辑。

### 10.9 用户目的不清晰

情况：

用户输入内容或目的说明不足以判断写作目标、分类、整理方式或是否允许扩写。

处理：

- 生成最多 3 个求证问题。
- 保留用户原始输入，不清空编辑框。
- 默认不创建文章。
- 用户回答后重新生成草稿方案。
- 用户选择“按当前理解继续”时，必须保留低置信度、风险提示和人工审核清单。

### 10.10 图片位置无法确认

情况：

用户要求保留图片位置，但原文中没有可识别的图片语法、文件名、截图标记或上下文位置。

处理：

- 在求证问题中询问图片应放在哪些段落后。
- 如果用户暂不补充，正文中只能放置明确的待确认占位。
- 不得编造图片说明、图片内容或图片路径。
- 创建前预览必须展示所有图片占位，供用户人工审核。

### 10.11 权限模式与操作风险冲突

情况：

用户选择了“替我审批”或“完全访问权限”，但本次操作存在高风险、低置信度、求证未完成、作用范围不明确、schema 校验失败或业务规则校验失败。

处理：

- 自动降级为“请求批准”。
- 展示降级原因。
- 保留 AI 生成结果和用户输入。
- 等待用户手动确认、修改或重新生成。
- 不允许为了满足高权限模式而跳过校验。

### 10.12 文章索引扫描失败

情况：

知识库稳定模式扫描本地文章时发生文件读取失败、Front Matter 解析失败或目录不存在。

处理：

- 展示失败原因。
- 返回已成功解析的文章数量。
- 列出失败文件路径。
- 单个文件失败不影响其他文章查询。
- 如果扫描目录不存在，提示检查本地博客目录配置。

### 10.13 知识库依据不足

情况：

用户提出具体内容问题，但本地文章索引和正文分块中没有找到足够依据。

处理：

- 明确回答“当前文章中没有找到明确依据”。
- 展示已检索的关键词或主题。
- 可推荐用户换一种问法或先创建相关笔记。
- 不得用模型通用知识伪装成用户文章内容。

### 10.14 本地索引损坏或过期

情况：

本地索引文件无法解析、内容 hash 与文章文件不一致，或索引时间早于文章修改时间。

处理：

- 自动重建索引。
- 重建失败时提示用户手动点击“重新扫描”。
- 重建期间不输出确定性知识库回答。

## 11. 验收标准

### 11.1 Agent 模式验收

- 用户能进入 AI 辅助写作页面。
- 用户能手动选择 Agent 模式。
- 用户发送前能选择“请求批准”“替我审批”“完全访问权限”，默认是“请求批准”。
- 用户输入杂乱笔记后，AI 能生成结构化草稿方案。
- 用户能用简单或详细自然语言说明本次目的，系统能识别为任务约束。
- 用户目的不清晰时，系统会向用户提出少量关键求证问题。
- 用户回答求证问题后，系统能基于新回答修正草稿方案。
- 用户可以选择按当前理解继续，但页面必须保留低置信度或风险提示。
- 用户只输入一句话想法时，AI 能生成短草稿和待补充问题，而不是编造完整长文。
- 用户复制外部笔记并要求规范整理时，AI 会保留原意、重排结构，并避免大幅虚构扩写。
- 用户要求保留图片位置时，正文预览中能看到对应图片占位。
- 用户输入学习笔记、踩坑复盘、项目日志、思考随笔时，AI 能识别不同写作目标并使用合适结构。
- 用户可以把输入先保存为待整理记录，不必须立即创建文章。
- 页面能展示分类、前缀、标题、标签、描述和正文预览。
- 页面能展示写作目标、用户目的理解、输入类型、大纲、求证问题、图片占位、待补充问题、风险提示和置信度。
- 页面能展示创建前人工审核清单。
- 请求批准模式下，用户未确认前不创建任何文章文件。
- 求证未处理且用户未确认按当前理解继续时，不允许创建草稿。
- 请求批准模式下，用户未点击创建前不创建草稿。
- 替我审批模式下，低风险且校验通过的草稿可以自动创建；高风险时必须降级请求批准。
- 完全访问权限下，可以在当前 Agent 任务内自动创建和写入草稿；但不能发布、提交 Git、删除文件或修改主题源码。
- 用户确认后，系统通过现有创建文章接口创建草稿。
- 新文章文件名符合 `prefix1-prefix2-title.md`。
- 新文章 Front Matter 符合项目规范。
- 新文章 `status` 为 `draft`。
- 创建成功后跳转编辑器。
- 创建失败时不丢失草稿方案。
- AI 不会选择分类注册表之外的一级分类。
- AI 不会把未经用户提供的经历、数据、版本号、路径或命令写成确定事实。
- 输入信息不足时，AI 会生成可继续完善的草稿，并明确提示待补充内容。
- 跨多个主题的输入不会自动创建多篇文章，只生成一个主方案并说明取舍。
- Agent 每次生成都有可追踪的 `sessionId`。
- 重复点击创建不会重复创建多篇相同文章。
- JSON 解析失败或校验失败时，不进入创建流程。
- 创建成功但写入失败时，仍保留已创建路径和用户输入。

### 11.2 LLM 模式验收

- 编辑器中可以展开和收起 AI 侧边栏。
- 侧边栏能读取当前文章内容。
- 用户发送前能选择三种权限，默认是“请求批准”。
- 有选中文本时，AI 修改默认作用于选区。
- 无选中文本时，插入类操作能插入到光标位置。
- 请求批准模式下，AI 生成结果不会自动写入文章。
- 请求批准模式下，用户点击采纳后才修改编辑器内容。
- 请求批准模式下，所有 LLM 修改都必须等待用户采纳。
- 替我审批模式下，低风险插入、替换和标题/描述优化可自动采纳，高风险操作仍需确认。
- 完全访问权限下，当前文章范围内的插入、替换、删除、全文重写和允许的 Front Matter 修改可自动执行，但必须记录操作摘要。
- 用户点击放弃后文章内容不变。
- 删除操作必须先展示删除摘要。
- 全文重写必须展示全文预览。
- 选区模式下不会修改选区之外的内容。
- 删除范围不明确时不会直接删除内容。
- 用户要求补充事实但上下文没有依据时，AI 不会编造事实。
- 用户要求发布、提交、删除文件或绕过确认时，AI 不会生成可执行写入操作。

### 11.3 知识库稳定模式验收

- 用户能手动选择知识库稳定模式。
- 该模式默认只读，不展示文章写入权限按钮。
- 用户询问“我有哪些文章”时，系统能扫描本地文章并返回准确文章清单。
- 用户询问文章总数时，系统返回的数量与本地 Markdown 文件扫描结果一致。
- 用户询问某个分类下的文章时，系统能按 Front Matter 或目录规则准确过滤。
- 用户询问某个主题是否写过时，系统能返回匹配文章和匹配依据。
- 用户询问具体文章内容时，系统回答中包含文章标题、路径和相关小节。
- 找不到依据时，系统明确说明没有找到，不编造答案。
- 本地索引缺失时可以自动重建。
- 本地索引损坏时不会导致页面崩溃。
- 本模式不依赖数据库服务。
- 本模式初始版本不要求引入向量数据库。
- 如果使用 embedding 缓存，缓存必须可删除、可重建，并与文章 hash 绑定。

### 11.4 统一模式与基础限制验收

- AI 辅助写作页面不因本地模式或在线模式展示不同主流程。
- Agent 创建草稿和 LLM 修改文章都通过统一接口完成。
- 前端不根据模式分支拼接不同的创建或保存逻辑。
- 三种权限模式都不能绕过 schema 校验、分类校验、Front Matter 校验和接口鉴权。
- “完全访问权限”不放行发布、部署、Git、删除文件、修改主题源码、创建未注册一级分类或系统配置修改。
- Agent 模式由 LangGraph 或 LangChain 工作流驱动。
- LLM 模式由 LangChain 或 LangGraph 生成结构化编辑操作。
- 知识库稳定模式由本地扫描、可重建索引和检索增强问答驱动。
- 未配置模型时不能生成。
- 非法 `prefix2` 不能创建文章。
- Agent 和 LLM 的输出都经过 schema 校验和业务规则校验后才允许进入预览或采纳流程。
