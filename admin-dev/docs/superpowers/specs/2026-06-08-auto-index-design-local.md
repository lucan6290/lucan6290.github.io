# 博客目录自动更新 — 文件系统设计

> 日期：2026-06-08
> 状态：设计中（部分待确认）
> 涉及项目：`blog/admin-dev`（管理后台 → Python FastAPI 后端）

---

## 一、背景

博客有一篇手动维护的目录文章 `source/_posts/index.md`（`slug: blog-index`，`sticky: 9999`），按一级/二级分类组织所有已发布文章的快捷链接。目前每次新建、删除、修改文章后，需要手动去 `index.md` 中增删改对应链接。

**目标**：在管理后台操作文章时，通过 Python FastAPI 后端自动同步更新 `index.md` 中的目录链接。

**架构说明**：

```
管理后台前端 (Vue 3 / Naive UI)
        │
  Python FastAPI
    (files.py)
        │
        ▼
   本地文件系统
```

- 管理端前端调用 Python FastAPI 后端，后端负责文件系统读写。

---

## 二、需求规则

| # | 规则 | 备注 |
|---|------|------|
| 1 | 文章发布时（status 变为 `published`），自动在 `index.md` 对应位置插入链接 | |
| 2 | 文章删除时，自动从 `index.md` 移除对应链接 | |
| 3 | 文章标题或日期修改时，同步更新目录中的链接文本和 URL | |
| 4 | 文章取消发布（`published` → `draft`/`wip`）时，从目录移除 | |
| 5 | 链接格式：`/{年}/{月}/{日}/{分类slug}/{文件名}/` | 年月日取自 `date` 字段 |
| 6 | 新文章链接用**尾插法**，追加到对应二级分类末尾 | |
| 7 | 如果二级分类在目录中不存在，自动创建带序号的标题（如 `### 1.4 xxx`） | 序号 = 该一级分类下最后一个二级分类序号 + 1 |
| 8 | 二级分类标题名取自 Front Matter `categories[1]` | |
| 9 | 删除文章后，如果某二级分类下文章清空，移除该标题 | 并对该一级分类下剩余二级分类重新编号 |
| 10 | 一级分类下文章全部清空时，恢复 `*暂无文章，敬请期待*` 占位文字 | 有文章时替换掉占位文字 |
| 11 | 状态变更：`published` → 非published 移除；非published → `published` 插入 | |

## 三、实现方案

### 3.1 架构

将目录更新逻辑抽象为**纯函数式引擎**（输入 `index.md` 文本 + 操作指令 → 输出修改后的文本），在 Python 端实现：

- **Python 端**：新建 `services/index_updater.py`

### 3.2 目录结构解析

将 `index.md` body 部分解析为结构化树：

```
IndexStructure:
  primary_categories: [
    {
      number: "一",                 # 中文序号
      title: "技术研习",
      heading: "## 一、技术研习",   # 原始标题行
      subcategories: [
        {
          number: "1.1",
          title: "AI 探索",
          heading: "### 1.1 AI 探索",
          articles: [
            "- [Agent 入门笔记](/2026/05/20/tech-study/ts-ai-agent入门笔记/)",
            "- [Claude Code 入门指南](/2026/05/22/tech-study/ts-claude-code入门与安装/)",
          ]
        },
        ...
      ],
      placeholder: null             # 有文章时为 null
    },
    {
      number: "三",
      title: "项目实战",
      subcategories: [],
      placeholder: "*暂无文章，敬请期待*"
    },
    ...
  ]
```

### 3.3 四个核心操作函数

#### ① add_to_index（插入文章链接）

```
输入: index文本, 文章metadata（title, date, categories, filename, category_slug）
1. 解析 index → IndexStructure
2. 用 categories[0] 匹配一级分类（中文分类名匹配标题）
3. 用 categories[1] 匹配二级分类：
   - 找到 → 尾插到该二级分类的 articles 末尾
   - 没找到 → 新建二级分类，序号 = 该一级分类下最大序号 + 1，追加到 subcategories 末尾
4. 如果一级分类有 placeholder，清除 placeholder
5. 序列化回 markdown 文本，返回
```

#### ② remove_from_index（移除文章链接）

```
输入: index文本, 文件名（用于匹配）
1. 解析 index → IndexStructure
2. 遍历所有一级/二级分类，按文件名匹配找到并移除对应链接
3. 如果某二级分类 articles 为空 → 移除该二级分类
4. 对该一级分类下剩余二级分类重新编号（如 1.1, 1.2, 1.3 → 移除 1.2 后变为 1.1, 1.2）
5. 如果某一级分类的所有二级分类都为空 → 恢复 "*暂无文章，敬请期待*"
6. 序列化回 markdown 文本，返回
```

#### ③ update_in_index（更新文章链接）

```
输入: index文本, 旧文件名, 新metadata（title, date, categories, category_slug）
1. 解析 index → IndexStructure
2. 用旧文件名找到旧链接
3. 对比新旧数据：
   - 只变了标题 → 更新链接文本 [新标题](...)
   - 变了日期 → 更新链接 URL 中的日期部分
   - 变了分类（categories 变化）→ remove(旧) + add(新)
4. 序列化回 markdown 文本，返回
```

#### ④ handle_status_change（状态变更处理）

```
输入: index文本, 文章metadata, 旧status, 新status
- published → 非published：调用 remove_from_index
- 非published → published：调用 add_to_index
- 其他情况（如 draft → wip）：不做处理
```

### 3.4 辅助函数

#### 链接生成

```
输入: 文章 date, category_slug, filename, title
date = "2026-06-08 15:30:00"
→ year = "2026", month = "06", day = "08"
→ 链接行: - [{title}](/{year}/{month}/{day}/{category_slug}/{filename}/)
```

#### 链接匹配

用**文件名**作为唯一标识匹配目录中的链接（文件名不变）：
```
正则: - \[.*?\]\(/.*?/({filename})/\)
```

#### 分类名匹配

一级分类匹配需要将 Front Matter 中的中文分类名映射到目录标题：
```
categories[0] = "技术研习" → 匹配 "## 一、技术研习"（用 contains 匹配）
```

#### 中文序号映射

```
1 → 一, 2 → 二, 3 → 三, 4 → 四, 5 → 五
```

### 3.5 序列化规则（IndexStructure → Markdown）

保持 `index.md` 的格式风格：
- 一级分类之间用 `---` 分隔
- 每个一级分类占位文字独占一行：`*暂无文章，敬请期待*`
- 文章链接格式：`- [标题](/URL/)`
- Front Matter 原样保留，只修改 body 部分
- 末尾保留 `*持续更新，记录成长*`
- `####` 层级的系列文章组原样保留，不做修改（不在本次范围内）

## 四、集成点

在 `backend/app/routers/files.py` 中集成：

| 路由 | 触发操作 | 调用函数 |
|------|----------|----------|
| `POST /post`（创建文章） | 文章创建后不触发（因为初始状态为 draft） | 无 |
| `PUT /post/{path}`（更新文章） | 检测 status/title/date/categories 变化 | `handle_status_change` / `update_in_index` |
| `DELETE /post/{path}`（删除文章） | 删除文章后 | `remove_from_index` |

**注意**：创建文章（`hexo np`）时初始状态为 `draft`，不触发目录更新。只有当用户将状态改为 `published` 并保存时（走 `PUT`），才通过 `handle_status_change` 插入目录。

## 五、错误处理

### 5.1 文件级错误

| 错误场景 | 处理策略 | 返回信息 |
|----------|----------|----------|
| `index.md` 文件不存在 | **终止操作**，记录错误日志，不自动创建 | `目录文件 index.md 不存在，请先手动创建` |
| `index.md` 内容为空（body 为空） | **终止操作**，不往空文件中写入 | `目录文件内容为空，请检查 index.md 格式` |
| `index.md` Front Matter 解析失败 | **终止操作**，保留原文件不变 | `index.md Front Matter 格式异常，无法解析` |
| 文件系统写入失败（权限不足等） | **终止操作**，抛出异常，原文件不变 | `写入 index.md 失败：{系统错误信息}` |

### 5.2 解析错误

| 错误场景 | 处理策略 | 说明 |
|----------|----------|------|
| 一级分类标题格式不符合 `## {中文序号}、{名称}` 模式 | **跳过该行**，继续解析下一行 | 记录警告日志：`跳过无法识别的标题行：{原文}` |
| 二级分类标题格式不符合 `### {数字}.{数字} {名称}` 模式 | **跳过该行**，其中内容归入上一个有效分类 | 记录警告日志 |
| `####` 系列文章组标题 | **跳过该行及下属内容**，原样保留在序列化输出中 | 系列文章不在本次范围内，不解析不修改 |
| 文章链接格式不符合 `- [{标题}](/{URL}/)` 模式 | **跳过该行**，原样保留在序列化输出中 | 不修改无法识别的行 |
| `index.md` 中完全没有一级分类标题 | **终止操作**，不修改文件 | 返回 `目录结构异常：未找到任何一级分类` |

### 5.3 匹配错误

| 错误场景 | 处理策略 | 说明 |
|----------|----------|------|
| 一级分类名 `categories[0]` 在目录中找不到匹配 | **自动追加**：在目录末尾创建新一级分类 | 中文序号 = 当前最大序号 + 1（如已有五则新增 "六"） |
| 二级分类名 `categories[1]` 在目录中找不到匹配 | **自动创建**（规则 7 已定义） | 序号 = 该一级分类下最大二级序号 + 0.1 |
| 用文件名匹配到**多条**链接 | **终止操作**，返回错误提示 | `目录中发现重复文件名 [{filename}]，请手动检查 index.md` |
| 用文件名匹配到**零条**链接（remove/update 时） | **静默跳过**，不报错，原文件不变 | 记录警告日志：`未在目录中找到文章 [{filename}] 的链接` |
| 文章 `categories` 字段为空或不存在 | **终止操作**，不插入 | `文章缺少 categories 字段，无法确定分类位置` |
| 文章 `categories` 只有一级、没有二级 | **使用一级分类名作为二级分类名** | 如 `categories: [技术研习]` → 在 "技术研习" 下创建 `### {N}.1 技术研习` |
| 文章 `date` 字段为空或格式异常 | **终止操作**，不插入 | `文章日期格式异常：{date}，无法生成链接` |
| 文章 `title` 字段为空 | **使用文件名作为标题** | 提取文件名中去掉前缀的部分作为回退标题 |

### 5.4 操作幂等性

| 场景 | 策略 | 说明 |
|------|------|------|
| `add` 时文章已在目录中存在 | **静默跳过**，不重复添加 | 记录信息日志：`文章 [{filename}] 已在目录中，跳过插入` |
| `remove` 时文章不在目录中 | **静默跳过**，不报错 | 见上方"匹配零条链接" |
| `update` 时新旧数据完全相同 | **跳过操作**，不修改文件 | 比对后无变化则直接返回原文本 |

### 5.5 编号边界

| 边界情况 | 处理策略 | 说明 |
|----------|----------|------|
| 一级分类超过 5 个（超出中文序号映射表） | **动态生成**，不依赖映射表 | 使用规则：`["一","二","三","四","五","六","七","八","九","十","十一",...]` |
| 二级分类编号 ≥ 10（如 `1.10`） | **正常编号**，不做特殊处理 | 允许 `1.10`、`2.11` 等多位数字编号 |
| 移除二级分类后重新编号时跳号 | **顺序重排**，消除间隙 | 如 `1.1, 1.3, 1.5` → 重排为 `1.1, 1.2, 1.3` |
| 所有二级分类被移除后一级分类为空 | **恢复占位符** `*暂无文章，敬请期待*` | 规则 10 已定义 |
| 所有一级分类都为空 | **正常保留**所有占位符，不删除一级分类 | |

## 六、边界情况

### 6.1 标题/文件名中的特殊字符

| 字符 | 场景 | 处理策略 |
|------|------|----------|
| 方括号 `[` `]` | 文章标题含 `[笔记]` 等 | 链接文本中方括号需原样保留：`- [Vue3 [笔记]](/...)`（Markdown 允许） |
| 半角括号 `(` `)` | 文章标题含括号 | 链接文本中正常使用，URL 中按 Hexo 生成规则保持原样 |
| 全角括号 `（` `）` | 标题中常见（如 `别再研究Prompt了，AI正在重构你的行业`） | 正常处理，不影响 Markdown 链接语法 |
| 中文斜杠 `／` | 极少出现在标题中 | 正常处理 |
| 冒号 `：` / `:` | 标题中常见 | 链接文本中正常使用 |
| 文件名含空格 | 理论上不会出现（`hexo np` 命名不含空格） | URL 中空格编码为 `%20`，匹配时需注意 |
| Emoji | 理论上不会出现在文件名中 | 如出现，正则匹配需支持 Unicode |

### 6.2 分类变更（跨分类移动）

| 场景 | 处理流程 | 说明 |
|------|----------|------|
| 一级分类变更（如 `技术研习` → `踩坑复盘`） | `remove`（旧位置）+ `add`（新位置） | 可能触发占位符/重编号逻辑 |
| 二级分类变更（如 `AI 探索` → `入门笔记`） | `remove`（旧二级）+ `add`（新二级） | 同上 |
| 一级 + 二级同时变更 | 同上，remove + add | |
| 变更后旧二级分类清空 | 自动移除该二级分类标题 + 重编号 | |
| 变更后旧一级分类清空 | 自动恢复占位符 | |

### 6.3 首次使用场景

| 场景 | 处理策略 | 说明 |
|------|----------|------|
| 目录中没有任何文章（全是占位符） | 第一篇 `add` 操作正常执行，替换对应一级分类的占位符 | |
| 新一级分类首次有文章 | 自动创建该一级分类的完整结构（`##` 标题 + `###` 子分类 + 链接） | 需正确计算中文序号 |
| 新二级分类首次有文章 | 自动创建 `###` 子分类标题 + 链接 | 需正确计算编号 |

### 6.4 性能边界

| 场景 | 当前策略 | 建议 |
|------|----------|------|
| `index.md` 文件过大（>100KB） | 每次操作全量解析 + 全量序列化 | 当前期规模（<50篇文章）无需优化；若超过 200 篇再考虑增量编辑 |
| 单次操作触发多步（如 remove + add） | 步骤间共享解析结果 | `update_in_index` 中分类变更场景，只解析一次，在内存中完成 remove + add 后序列化一次 |

### 6.5 数据一致性

| 场景 | 风险 | 保护策略 |
|------|------|----------|
| 操作中断（进程崩溃/网络断开） | index.md 可能处于半修改状态 | **写前备份**：操作前保存 `index.md.bak`；写后验证：序列化结果重新解析确认结构完整；确认成功后删除备份 |
| 两次快速连续操作同一篇文章 | 第二次操作可能覆盖第一次 | **文章级锁**：使用文件名作为锁 key，同一文章的操作排队执行 |
| index.md 被用户手动编辑 | 结构可能与预期不符 | 解析时容忍未知格式行（跳过而非报错），只修改能识别的部分 |
| Front Matter 中 `status` 字段缺失 | 无法判断是否应出现在目录中 | **默认不在目录中**，只有明确 `status: published` 才触发 add |
